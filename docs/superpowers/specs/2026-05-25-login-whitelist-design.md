# 用户白名单设计

## 背景

当前系统支持企业微信 SSO/扫码登录，并在登录时同步企业微信用户姓名、部门、职位等信息。管理员侧边栏已有多个管理入口，但缺少控制“哪些企业微信用户可以登录本系统”的能力。

本次新增用户白名单功能，允许管理员从用户姓名和部门两个维度配置登录准入规则。白名单仅限制企业微信登录用户，本地账号不受限制，保留管理员兜底能力。

## 目标

- 管理员可以在侧边栏进入“用户白名单”页面。
- 管理员可以添加、查看、删除用户姓名白名单。
- 管理员可以通过部门名称模糊搜索企业微信部门，并添加、查看、删除部门白名单。
- 企业微信登录时按白名单判断是否允许进入系统。
- 部门白名单包含该部门及其所有多层级后代部门成员。
- 未配置任何用户或部门白名单时，不限制企业微信用户登录。

## 非目标

- 不做批量导入。
- 不做白名单启停开关。
- 不做备注、审批流或审计日志。
- 不限制本地账号登录。

## 规则定义

### 用户姓名白名单

管理员配置企业微信用户姓名。保存前后端均去除前后空格，空字符串不允许保存。

登录匹配时使用完全相等匹配。例如配置 `张三` 时，`张三` 可以登录，`张三天` 不可以登录。

### 部门白名单

管理员通过模糊搜索部门名称选择部门，系统保存企业微信部门 ID，而不是部门名称。展示时返回部门名称和路径，避免重名部门造成歧义。

登录匹配时，如果用户所属任一部门本身在白名单中，或其任一祖先部门在白名单中，则允许登录。也就是说配置 A 部门后，A 部门及 A 的所有多层级后代部门成员均可登录。

### 组合规则

用户姓名白名单和部门白名单为“或”关系，任一命中即可登录。

当用户姓名白名单和部门白名单均为空时，企业微信登录不受限制。

## 后端设计

### 数据模型

新增两张表：

`login_whitelist_users`

- `id`：自增主键。
- `name`：去除前后空格后的企业微信姓名，唯一且非空。
- `created_at`：创建时间。

`login_whitelist_departments`

- `id`：自增主键。
- `department_id`：企业微信部门 ID，唯一且非空，关联现有 `departments.id` 的业务语义。
- `created_at`：创建时间。

数据库初始化遵循现有 `backend/app/db/migrations.py` 的兼容迁移模式，在应用启动时补齐表结构。

### 服务模块

新增独立白名单服务模块，封装以下职责：

- 获取当前白名单配置。
- 搜索部门并生成部门路径。
- 判断企业微信用户详情是否允许登录。
- 根据现有 `departments(id, name, parent_id)` 构建部门父子/祖先关系。

部门判断建议一次性读取部门表，在内存中沿 `parent_id` 向上追溯用户部门的祖先，判断是否命中白名单部门集合。这样可以支持多层级后代部门，并避免把递归逻辑散落在 API 路由中。

### 管理 API

新增管理员接口，统一放在 `/api/admin/login-whitelist` 下，并使用现有 `require_admin` 保护。

- `GET /api/admin/login-whitelist`
  - 返回用户姓名白名单和部门白名单。
  - 部门项包含 `id`、`department_id`、`name`、`path`。
- `POST /api/admin/login-whitelist/users`
  - 添加用户姓名。
  - 后端 trim；空字符串返回 400；重复返回 409。
- `DELETE /api/admin/login-whitelist/users/{id}`
  - 删除用户姓名白名单。
- `GET /api/admin/login-whitelist/departments/search?q=关键词`
  - 按部门名称模糊搜索。
  - 返回匹配部门的 `department_id`、`name`、`path`。
- `POST /api/admin/login-whitelist/departments`
  - 按 `department_id` 添加部门白名单。
  - 部门不存在返回 404；重复返回 409。
- `DELETE /api/admin/login-whitelist/departments/{id}`
  - 删除部门白名单。

### 登录链路

企业微信 SSO 和二维码登录都在获取企业微信用户详情后、创建或更新本地用户前执行白名单校验。

校验流程：

1. 查询用户姓名白名单和部门白名单。
2. 如果两类白名单均为空，允许登录。
3. 使用企业微信姓名 trim 后与姓名白名单做完全相等匹配。
4. 使用企业微信返回的部门 ID 列表做部门和祖先部门匹配。
5. 任一命中则允许登录，否则拒绝。

拒绝时：

- SSO 回调重定向到 `/login?error=login_whitelist_denied`。
- 二维码登录接口返回 403，错误文案为“当前账号未在登录白名单中，请联系管理员”。
- 不创建或更新本地用户，不创建工作区，不写入 session。
- 后端记录 warning 日志，包含企微 `userid`、姓名和部门 ID，便于排查。

本地账号不走企业微信登录链路，因此不受白名单限制。

## 前端设计

### 侧边栏入口

`frontend/src/components/Sidebar.tsx` 在管理员“管理”分组下新增“用户白名单”入口，仅 `user.role === "admin"` 可见。

`ViewName` 增加 `loginWhitelist`，`App.tsx` 增加页面渲染和面包屑。

### 管理页面

新增 `LoginWhitelistPage`，风格保持与现有管理页面一致。

页面包含：

- 标题“用户白名单”和刷新按钮。
- 用户姓名配置区域：
  - 输入框。
  - 添加按钮。
  - 已配置姓名列表。
  - 每项支持删除。
- 部门配置区域：
  - 搜索框。
  - 搜索结果展示部门路径。
  - 点击搜索结果添加部门。
  - 已配置部门列表展示部门路径。
  - 每项支持删除。

状态处理：

- 加载中、空状态和接口错误使用现有 UI 组件风格。
- 添加成功、删除成功、重复添加、权限失败等使用现有 `useToast` 提示。
- 部门搜索关键词为空时不请求或返回空结果。

### 登录页错误提示

登录页识别以下情况并展示“当前账号未在登录白名单中，请联系管理员”：

- URL 参数 `error=login_whitelist_denied`。
- 二维码登录接口返回 403。

## 测试设计

### 后端测试

- 空白名单时允许企业微信登录。
- 用户姓名 trim 后完全相等才允许；`张三` 不允许 `张三天`。
- 配置部门 A 时，A 本部门用户允许。
- 配置部门 A 时，A 的多层后代部门用户允许。
- 配置部门 A 时，A 的父部门、兄弟部门用户不允许。
- 同时配置姓名和部门时，任一命中即可登录。
- 未命中白名单时，SSO 不创建 session，二维码登录返回 403。
- 管理员接口要求 admin 权限。
- 添加空姓名返回 400，重复姓名返回 409。
- 添加不存在部门返回 404，重复部门返回 409。
- 部门搜索支持模糊关键词，并返回部门路径。

### 前端验证

- TypeScript 构建通过。
- 管理员可见“用户白名单”入口，普通用户不可见。
- 页面可加载、添加、删除用户姓名白名单。
- 页面可搜索、添加、删除部门白名单。
- 登录页能展示白名单拒绝文案。

## 风险与取舍

- 部门表依赖企业微信同步结果。如果部门未同步，部门搜索无法找到目标部门，登录时也无法正确判断后代关系。该行为符合当前系统的数据依赖，后续可通过同步状态提示增强。
- 部门重名通过保存部门 ID 和展示路径解决。
- 不做启停开关，避免“空白名单”和“关闭限制”语义冲突。当前规则明确为空白名单时不限制登录。
- 本地账号不受限制，避免白名单配置错误导致管理员无法恢复配置。
