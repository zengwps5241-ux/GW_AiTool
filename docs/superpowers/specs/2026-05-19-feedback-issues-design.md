# 问题反馈功能设计

## 背景

当前系统缺少面向用户的问题反馈入口。用户在使用对话、工作空间、插件技能等功能时，无法直接提交带上下文描述和截图的问题；管理员也没有统一的反馈管理页面查看和跟进反馈。

本次功能新增顶部反馈入口和管理员反馈管理页。普通用户可以提交标题、详细描述和粘贴图片；管理员可以分页查看反馈列表，并进入详情查看文字和图片附件。

## 目标

- 在顶部栏主题切换按钮左侧增加反馈图标入口。
- 用户点击反馈图标后弹出反馈面板。
- 反馈面板支持输入问题标题、详细描述，并在描述区域粘贴图片。
- 粘贴的图片作为附件持久化保存，提交前可预览和删除。
- 管理员侧边栏新增“反馈管理”入口。
- 管理员可分页查看反馈列表，列表显示标题、提出人、提出时间。
- 管理员点击标题进入详情，查看标题、描述、提出人、提出时间和图片附件。

## 非目标

- 不做普通用户的“我的反馈”列表。
- 不做反馈状态流转、评论、分派、关闭等 issue tracker 能力。
- 不采集浏览器环境、页面路径、会话内容等额外诊断信息。
- 不支持非图片附件。
- 不对图片做 OCR、压缩转码或缩略图生成。

## 后端设计

### 数据模型

新增 `FeedbackIssue`：

- `id: int`
- `title: str`
- `description: str`
- `reporter_username: str`
- `created_at: datetime`

新增 `FeedbackAttachment`：

- `id: int`
- `issue_id: int`
- `filename: str`
- `content_type: str`
- `file_path: str`
- `size: int`
- `created_at: datetime`

表名使用：

- `feedback_issues`
- `feedback_attachments`

迁移沿用现有 `backend/app/db/migrations.py` 的轻量初始化方式，在 `init_db()` 中创建缺失表。

### 文件存储

图片附件不进入用户工作空间，避免和用户文件管理混在一起。

默认落盘目录：

```text
feedback_uploads/<issue_id>/<attachment_id-or-safe-name>
```

文件名需要经过安全清洗；实际持久化路径由后端生成，不能信任浏览器传入的文件名。数据库中保存相对 `feedback_uploads` 根目录的路径。

### 接口

`POST /api/feedback/issues`

- 登录用户可访问。
- 请求类型：`multipart/form-data`
- 字段：
  - `title`
  - `description`
  - `images`
- 校验：
  - 标题不能为空，限制最大长度。
  - 描述允许为空但整体反馈需要标题。
  - `images` 只允许 `image/*`。
  - 单张图片和总大小设置上限。
- 返回：创建后的 issue 基础信息。

`GET /api/admin/feedback/issues?page=1&page_size=20`

- 仅管理员。
- 返回分页结果：
  - `items`
  - `total`
  - `page`
  - `page_size`
- 每项包含：
  - `id`
  - `title`
  - `reporter_username`
  - `created_at`

`GET /api/admin/feedback/issues/{id}`

- 仅管理员。
- 返回详情：
  - `id`
  - `title`
  - `description`
  - `reporter_username`
  - `created_at`
  - `attachments`

`GET /api/admin/feedback/attachments/{id}`

- 仅管理员。
- 返回图片文件。
- 如果附件不存在或文件缺失，返回 404。

### 权限

- 提交反馈：所有已登录用户。
- 管理反馈：仅管理员。
- 查看图片附件：仅管理员。

## 前端设计

### 顶部反馈入口

`Topbar` 在主题切换图标左侧增加反馈图标按钮。图标使用现有 icon 体系中合适的反馈/消息/告警类图标。

点击后打开反馈面板。

### 反馈面板

反馈面板包含：

- 标题输入框。
- 详细描述 textarea。
- 图片附件预览区。
- 提交和取消按钮。

粘贴图片行为：

- 在 textarea 中监听 paste 事件。
- 遍历剪贴板 items，提取 `image/*` 文件。
- 图片加入本地附件列表并展示预览。
- 提交前允许删除单张图片。

提交行为：

- 使用 `FormData` 调用 `POST /api/feedback/issues`。
- 提交中禁用按钮。
- 成功后 toast 提示并关闭面板。
- 失败后 toast 展示错误。

### 管理页面

侧边栏管理员可见新增“反馈管理”入口。

新增 `FeedbackPage`：

- 默认显示列表视图。
- 列表列：标题、提出人、提出时间。
- 分页按钮：上一页、下一页。
- 点击标题加载详情并切换到详情视图。
- 详情视图展示标题、描述、提出人、提出时间和图片附件。
- 图片通过管理员附件接口加载，直接在详情页展示。

## 数据流

1. 用户点击顶部反馈图标。
2. 用户输入标题和描述，粘贴图片。
3. 前端用 `FormData` 提交标题、描述和图片文件。
4. 后端创建 issue 记录。
5. 后端保存图片文件并创建 attachment 记录。
6. 管理员进入反馈管理，分页拉取 issue 列表。
7. 管理员点击标题，拉取详情和附件元数据。
8. 详情页通过附件接口加载图片。

## 错误处理

- 标题为空：前端拦截，后端也返回 400。
- 图片类型不合法：后端返回 400，前端 toast 展示。
- 图片过大：后端返回 400，前端 toast 展示。
- 附件保存失败：后端应回滚本次 issue 创建，避免出现无效详情。
- 管理员接口无权限：返回 403。
- 附件文件缺失：返回 404，详情页显示图片加载失败。

## 测试计划

后端：

- 普通登录用户可提交文字反馈。
- 普通登录用户可提交带图片反馈。
- 非图片附件被拒绝。
- 管理员可分页查看反馈列表。
- 管理员可查看反馈详情。
- 管理员可访问图片附件。
- 普通用户访问管理接口返回 403。

前端：

- `npm run build` 编译通过。
- 反馈面板的状态、粘贴图片附件、提交入口在类型层面通过构建验证。
- 管理页面路由、API client 和组件编译通过。

## 风险与约束

- 当前项目依赖本地 PostgreSQL 测试库，部分接口集成测试在本地数据库缺失时会被环境阻塞。
- 图片暂不做压缩，单张和总大小限制需要保守设置，避免磁盘被滥用。
- 附件 URL 只开放管理员接口，不能用静态目录直接暴露。
