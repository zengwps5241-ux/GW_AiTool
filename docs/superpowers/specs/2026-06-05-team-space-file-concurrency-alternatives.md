# 团队空间文件并发控制替代技术路线

生成日期：2026-06-05
状态：Draft
关联方案：`docs/superpowers/specs/2026-06-04-team-space-file-version-optimistic-lock-design.md`

## 1. 结论

文件乐观锁能解决一部分覆盖问题，但它不是最完美的团队空间并发模型。

它的问题不在版本字段，而在写入模型：如果前端、上传任务、转换任务、Agent `Write/Edit`、Agent `Bash` 都可能直接写共享工作区，那么系统只能在写入前做拦截和校验。只要有入口绕过校验，或者校验和真实写入之间不是同一个原子操作，就仍然可能出现并发写问题。

更好的方向是：

1. 共享工作区只允许一个受控写入通道修改文件。
2. 用户前端保存、上传、转换输出都提交给这个写入通道。
3. Agent 不直接写共享工作区，而是在隔离副本中读写，最后提交变更集。
4. 后端按顺序把变更集应用到共享工作区；能合并则合并，不能合并则报冲突。

一句话：不要让多个执行者同时碰共享目录。共享目录只接受串行化的提交。

## 2. 为什么当前乐观锁不完美

当前乐观锁路线是：

```text
读取文件版本 -> 编辑 -> 写入前校验版本 -> 版本一致则写入
```

这能发现“我读完之后别人改过文件”的情况，但仍有三个问题：

1. 写入入口多。前端 API 可以校验，Agent hook 可以校验，但 Bash、转换任务、上传覆盖等入口都必须完全接入，否则会绕过。
2. hook 不是文件系统。Agent 的 Bash 命令如果能直接写共享目录，hook 只能在执行前判断意图，不能天然保证真实写入过程可控。
3. 写前校验不是最终答案。真正安全的是“校验和写入在同一个受控临界区内完成”，否则两个写请求可能都通过校验。

所以更完美的设计不是继续增强 `FileVersion`，而是收敛写入路径。

## 3. 可选路线

### 方案 A：乐观锁 + 服务端内部写入锁

所有写入口仍然直接写共享工作区，但后端在每个目标路径上加内部锁：

```text
获取文件写入锁 -> 校验版本 -> 写入文件 -> 释放锁
```

优点：

- 改动最小。
- 前端编辑保存很容易接入。
- 对上传、转换、普通 API 写入也适用。

缺点：

- 仍然要求所有写入口都必须走受控 helper。
- Agent Bash 直写共享目录仍然是风险点。
- 冲突处理仍是“发现冲突后失败”，不是“隔离工作后合并”。

判断：这是当前方案的必要补丁，但不是更完美路线。

### 方案 B：共享工作区单写入通道

引入一个统一的文件写入服务，所有共享工作区变更都通过它提交：

```text
前端 / 上传 / 转换 / Agent
        |
        v
FileMutationService
        |
        v
按 workspace + path 串行执行写入
        |
        v
共享工作区
```

服务只暴露明确操作：

- create file
- update file
- delete file
- move file
- rename file
- apply patch

每个操作在服务内部完成权限校验、版本校验、路径锁、写入和返回新版本。

优点：

- 共享工作区只有一个可信写入入口。
- 并发写被服务端串行化，两个请求不会同时写同一文件。
- 可以复用现有前端文件 API。

缺点：

- Agent 不能再直接写共享目录。
- 需要改造 Agent 写工具或运行环境。
- Bash 写文件必须被限制或重定向到受控写入服务。

判断：这是前端和普通任务写入的正确方向，但 Agent 还需要隔离执行模型配合。

### 方案 C：Agent 隔离工作区 + 变更集提交

Agent run 不再直接使用团队空间目录作为可写 cwd。启动 Agent 时创建隔离工作区：

```text
共享工作区快照
        |
        v
Agent 隔离工作区
        |
        v
Agent 任意 Read / Write / Bash
        |
        v
生成变更集 ChangeSet
        |
        v
提交给 FileMutationService
        |
        v
串行应用到共享工作区
```

隔离工作区可以用：

- 临时目录复制。
- overlayfs。
- git worktree。
- rsync 快照。

Agent 在隔离区内可以自由使用 `Write/Edit/Bash`。它不会直接破坏共享工作区。Agent 结束后，后端计算变更集：

```json
{
  "base_revision": "workspace-rev-123",
  "changes": [
    {
      "path": "docs/a.md",
      "type": "modify",
      "base_sha256": "...",
      "new_sha256": "...",
      "content": "..."
    }
  ]
}
```

提交时，后端按顺序应用：

1. 校验用户仍有写权限。
2. 校验共享工作区当前文件是否仍等于 `base_sha256`。
3. 一致则应用变更。
4. 不一致则尝试三方合并。
5. 无法合并则返回冲突。

优点：

- Agent 的读写行为和共享工作区解耦。
- Bash 写入风险大幅下降，因为 Bash 只写隔离区。
- 可以支持 Agent 一次修改多个文件，然后作为一组变更提交。
- 用户能看到“Agent 准备改哪些文件”，后续可扩展审批。

缺点：

- 实现复杂度高于乐观锁。
- 需要计算变更集。
- 大工作区复制成本需要优化。
- 二进制文件和删除/移动操作需要单独处理。

判断：这是 Agent 场景下更完整、更稳的路线。

### 方案 D：Git 化工作区

把每个团队空间工作区变成 Git 仓库。每个用户或 Agent run 在独立分支/worktree 上工作，最终通过 merge 提交到主分支。

```text
main: 共享工作区
agent-run-1: Agent 分支
user-edit-1: 用户编辑分支
        |
        v
merge / conflict
```

优点：

- 天然有 base、diff、merge、冲突记录。
- 历史追踪和回滚能力强。
- Agent 变更集可以对应 commit。

缺点：

- 把文件系统协作问题升级成 Git 产品能力。
- 非技术用户不应该感知 Git 概念。
- 大文件、二进制、Office/PDF 转换产物会带来仓库膨胀。
- 实现和运维复杂度高。

判断：长期可以考虑，但首版不建议把团队空间直接 Git 化。

### 方案 E：实时协同 CRDT/OT

对文本文件使用 CRDT 或 OT 做多人实时协同编辑。

优点：

- 对同一文本文件多人同时编辑体验最好。
- 可以做到接近 Google Docs 的实时协同。

缺点：

- 只适合文本编辑器，不适合任意文件、Bash、上传、转换任务。
- 对 Agent 写文件不是天然适配。
- 工程复杂度远超当前需求。

判断：不是当前问题的合适解。

## 4. 推荐路线

推荐采用“单写入通道 + Agent 隔离工作区 + 变更集提交”的混合路线。

### 4.1 用户编辑

用户前端编辑仍然保持简单：

```text
打开文件 -> 保存版本 -> 编辑 -> 提交保存
```

但保存请求不直接写文件，而是提交给 `FileMutationService`：

```text
前端保存
    |
    v
FileMutationService.update_file()
    |
    v
内部串行校验和写入
    |
    v
共享工作区
```

用户体验仍然是保存成功或提示冲突，不引入可见锁。

### 4.2 Agent 编辑

Agent 不直接写共享工作区：

```text
创建 Agent 隔离工作区
        |
        v
Agent 在隔离区读写文件
        |
        v
Agent run 结束后生成 ChangeSet
        |
        v
FileMutationService.apply_changeset()
        |
        v
共享工作区
```

Agent 可以继续用 `Read/Write/Edit/Bash`，但它的 cwd 是隔离区。共享工作区对 Agent 是只读快照来源，不是直接写入目标。

### 4.3 冲突处理

提交变更集时出现冲突：

- 前端用户保存冲突：提示用户编辑冲突。
- Agent 提交冲突：提示 Agent 重新读取最新共享文件后再生成变更，或把冲突交给用户确认。

首版不强制做自动合并。只要不覆盖共享工作区即可。

## 5. 为什么这比单纯乐观锁更好

| 对比点 | 单纯乐观锁 | 推荐路线 |
| --- | --- | --- |
| 前端保存 | 写前校验 | 受控写入服务串行校验和写入 |
| Agent 写文件 | hook 拦截直接写共享目录 | Agent 只写隔离区 |
| Bash 风险 | 需要准确识别写命令 | Bash 只能影响隔离区 |
| 多文件修改 | 多个文件分别校验 | 一个 ChangeSet 统一提交 |
| 并发写 | 依赖所有入口正确校验 | 共享工作区只有一个写入口 |
| 后续扩展 | 主要是冲突失败 | 可扩展审批、diff、合并、回滚 |

核心变化是：从“多人直接写共享目录，靠校验阻止覆盖”，变成“多人只提交变更，后端串行应用变更”。

## 6. 首版落地建议

分两步做，避免一次性重构过大。

### 阶段 1：先收敛共享写入口

1. 建 `FileMutationService`。
2. 前端保存、新建、上传覆盖、转换输出都走这个服务。
3. 服务内部完成权限校验、版本校验、路径锁和写入。
4. 保留现有乐观锁字段，作为写入服务的校验条件。

这一步先解决普通用户和任务并发写。

### 阶段 2：Agent 改为隔离工作区

1. Agent run 启动时创建隔离工作区。
2. Agent cwd 指向隔离区。
3. 共享工作区内容按需复制或快照到隔离区。
4. Agent run 结束后计算变更集。
5. 变更集提交给 `FileMutationService`。

这一步解决 Agent/Bash 直接写共享目录的问题。

## 7. 最终判断

如果目标只是“尽快避免大多数覆盖”，文件乐观锁加内部写入锁够用。

如果目标是“更完美、更长期的团队空间文件协作”，推荐路线应该是：

```text
共享工作区单写入通道
        +
Agent 隔离工作区
        +
变更集串行提交
```

这条路线牺牲了一些实现复杂度，但换来更清晰的系统边界：共享工作区不再被多个执行者直接写，所有变更都以受控提交的方式进入。并发控制从“到处拦截写入”变成“只允许一个地方写入”。
