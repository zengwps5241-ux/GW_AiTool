优先级判断维度

从五个维度评估每个功能的优先级：

维度含义
依赖关系是否是其他功能的前置条件（基石不建，上层无法构建）
业务紧迫性V0.1（2-4周）需支撑实训营，V0.2（4-8周）需支撑首批 FDE 试点
频次与刚需每日高频使用 vs 偶尔触发；缺失即阻塞 vs 缺失可绕行
风险消除§7.4 假设栏中标记的高风险项应优先验证
闭环完整性是否构成"业务可走通"的最小闭环，避免功能孤岛
2

四级优先级分层

🔴 P0 — 平台基石
必须最早完成，V0.1 必交付
#功能模块对应 KR核心理由
1个人/团队空间文件管理KR1所有上层能力的"原料入口"。没有文件就没有 RAG、没有 Agent 上下文、没有资产沉淀
2基础权限治理
（白名单 + 空间隔离）		KR6§7.4 明确指出"权限和空间隔离是客户资料进入平台的前提"。晚做即返工
3Agent 中心 + 单 Agent 对话—平台的核心交付物。没有 Agent，平台就只是文件夹
4多任务对话（基础版）KR3FDE 的主工作台。一个 FDE 一天会并行 3-5 个调试任务
5/command 搜索 + 技能/插件中心KR4Agent 的"手脚"发现入口。无此则技能资产看不到
6文件预览 + 拖拽引用 + 反馈入口—高频交互动作，缺失会让用户立刻流失
🎯 P0 验证目标：FDE 能在 2 周内独立完成"上传文件 → 配 Agent → 调试对话 → 反馈"全链路（首版 Agent 通过文件直接引用获取上下文，RAG 调整至 P2）。
🟠 P1 — 闭环关键
V0.2 试点必交付
#功能模块对应 KR核心理由
1工具接入管理
（MCP Server/HTTP API 注册）		KR13Agent 调用真实业务系统的唯一通路。先于多智能体编排——单 Agent 都接不了工具，多 Agent 协作无意义
2数据工作台
（MinerU + ETL + Supabase）		KR10试点项目 80% 的工作是"先把客户数据搞进来"。无此则 FDE 退回 Excel+脚本
3知识图谱 + 本体工作台（基础）—"文件→数据→Agent"主循环的语义层。结构化数据需要本体/关系才能被 Agent 准确消费，避免每个 Agent 重复理解业务 schema
4Agent 业务视图
（表格 + 任务状态渲染）		KR11业务用户接触平台的唯一窗口。试点能否成功取决于此
5反馈与改进任务台 + 反馈回链KR9闭环"流程三"的承载。没有它，业务反馈就是聊天截图
6完整权限矩阵 + 审计日志KR6 增强客户私有化部署的合规前提
🎯 P1 验证目标：至少 1 个 FDE 小队在 V0.2 跑通 "文件 → 数据 → 知识图谱/本体 → Agent → 调用业务工具 → 业务用户使用 → 反馈回流" 主循环。
🟡 P2 — 规模化与复用
V1.0 BU 移交版
#功能模块优先级理由
1RAG 知识库 + 引用溯源 + 评测面板由原 P0 调整至 P2。RAG 本身工程量大且效果不稳定（§7.4 风险栏），V0.1 可让 Agent 通过文件直接引用获取上下文，V0.2 通过知识图谱/本体提供结构化语义层，待数据治理和评测样本就绪后再上 RAG
2团队空间 Release 发布 + 项目资产库由原 P1 调整至 P2。V0.2 先聚焦"文件→数据→Agent"主循环跑通，Release 沉淀放到 V1.0 阶段——MVP 还没成熟时发布快照只是空壳
3资产复用 Fork + 复用次数统计跟随 Release 同步下沉。Fork 是 Release 的消费端，必须等 V1.0 资产库有内容后才有意义
4多智能体团队编排（串行/并行）KR12 虽列入首版，但实操依赖单 Agent + 工具成熟。强行前置会陷入"多 Agent 互相调试"陷阱
5快照对比 + 行业/标签分类资产库规模到 20+ 后才有价值
6运营统计后台完整版复用率指标需要 P2 复用数据沉淀后才有意义
7Agent 配置复制和迁移跨团队复用的体验优化
8结构化输出扩展（看板/流程图）P1 已覆盖表格+任务状态，已能跑通试点
9表单驱动对话 + 推理过程摘要折叠体验加分项，非阻塞项
10标准交付包 + 部署清单 + SLABU 移交的标准化前提，但需 P1 试点经验沉淀后才知道"包"长什么样
🎯 P2 验证目标：至少 1 个成熟 MVP 完成 "Release 发布 → 另一小队 Fork → 客户定制" 的跨团队复用链路，同时沉淀首批 RAG 评测样本。
🟢 P3 — 后续扩展
V1.0 之后
  • 企微协同 + 群聊 Agent — 渠道扩展，非核心闭环
  • 定时任务 — 运营优化，可用外部 cron 替代
  • GraphRAG / 跨知识库推理 — 技术前沿，需 P2 知识图谱成熟
  • 动态业务工具开发 — FDE 工具箱沉淀，依赖 P1 工具接入成熟
  • 客户侧私有化部署模式 — §10 Open Question 1 待定，先内部部署验证
  • 跨业务线模板库 — 资产规模 50+ 后才有提炼空间

三个常见反模式（需主动避免）

❌
先做多 Agent 编排再做工具接入
Agent 没有手脚，多个 Agent 也只能互相聊天
❌
先做 Release/Fork 再做基础数据治理
MVP 没成熟就发布快照，资产库里全是空壳，Fork 出来也用不起来
❌
跳过本体/知识图谱直接上 RAG
Agent 拿到的是松散的文档片段，理解业务 schema 时反复出错——本体先行才能让结构化数据真正被消费

对 PRD 的几点建议

基于"Think Before Coding"原则，以下几处需要在动工前澄清：

4.1 V0.1 范围似乎略乐观（2-4 周）

P0 第 1-6 项全部要求 V0.1 交付，但其中 Agent 中心 + 多任务对话任一项的工程量都不止 1 周。建议：

  • RAG 知识库整体后置至 P2/V1.0（本计划已调整 ✅）：V0.1 阶段 Agent 通过文件直接引用获取上下文，待数据治理和评测样本就绪后再启动 RAG
  • 将 /command 跨技能包检索 简化为首版只支持本团队技能
  • Agent 业务视图 的 P1 范围可以仅做"消费 Markdown + 表格"，避免和对话工作台并行开发
4.2 P1 知识图谱/本体的"轻量起步"边界要先约定

本计划把"知识图谱+本体（基础）"升入 P1，作为"文件→数据→Agent"主循环的语义层。但本体建模容易陷入"建模炸弹"——一开始就追求完整的领域 schema 会拖慢 V0.2 全部进度。

建议：

  • V0.2 阶段只交付"最小可用本体"：每个试点场景定义 ≤ 10 个核心实体 + ≤ 20 个关系，覆盖 Agent 当前需要回答的问题即可
  • 本体编辑器只做表单/CSV 录入，不做可视化建模工具（推迟到 P2）
  • 知识图谱查询只支持基于本体 schema 的过滤查询，不做 GraphRAG 推理（P3 才做）
  • 这样可避免与"工具接入、数据工作台、业务视图"4 个核心模块并行 4-8 周时出现进度风险
4.3 §10 Open Question 应在 V0.1 启动前解决

特别是这三个，直接影响架构：

  • Q1：部署模式（内部 vs 客户侧） → 决定 Supabase 选型
  • Q2：Supabase 是终态还是临时方案 → 决定是否需要抽象数据层
  • Q5：权限模型是否已有现成设计 → 决定 P0 第 2 项的工期
4.4 缺一个"运营仪表盘最小可见"

KR8 要求"统计资产复用率"，但 PRD 中运营统计后台被排到 V1.0。建议：V0.2 至少有一个最简的复用计数页面（哪怕只有 1 个数字），否则无法验证 KR8。

最终优先级一句话总结

先把 "FDE 一个人能跑通 文件→Agent 主循环" 做扎实（P0），再把 "文件→数据→知识图谱→Agent→业务用户→反馈" 完整链路打通（P1），最后才考虑 "Release/Fork 沉淀复用" 和 "RAG / 多 Agent 等高级能力"（P2/P3）。

⚠️ 关键风险点：权限要早做、工具接入要早于多 Agent、本体/知识图谱先于 RAG、Release/Fork 属于 V1.0 不是 V0.2（MVP 未成熟时发布快照只是空壳）。
