## 0. v2 评审结论与改造原则

### 0.1 总体评价

原 v1 方案的方向是对的。它抓住了多智能体系统落地中的真实痛点：静态拓扑、上下文膨胀、质量不可控、人类难以结构化介入。`team.yaml` 和 `plan.yaml` 的声明式设计也很有价值，适合沉淀为企业级交付资产。

但 v1 仍然更像“产品理念 + 架构草图”。如果直接按 v1 实现，最容易出问题的地方是：

1. `plan.yaml` 同时承担计划定义和运行时状态，容易造成状态竞争与审计困难。
2. `Planning Agent` 权限过大，既生成计划、又验证产物、又能修改计划，是系统级单点风险。
3. 质量网关缺少强类型验证协议，容易退化成“让另一个 LLM 看一眼”。
4. 动态修改计划缺少版本化、审批和影响分析，会削弱用户掌控感。
5. 安全边界只写到 Docker/WASM，不足以覆盖 Prompt Injection、工具权限、密钥、网络、Artifact 污染等企业风险。
6. 调度模型缺少租约、并发锁、失败恢复、预算控制和可观测性。

### 0.2 v2 核心改造原则

v2 的核心原则是：

> YAML 声明期望状态，编排器维护实际状态，LLM 只提出建议，确定性系统负责执行、权限、验证、版本、审计。

这意味着：

| v1 思路 | v2 改造 |
|---|---|
| `plan.yaml` 保存计划和执行状态 | `plan.yaml` 只保存期望状态，运行状态进入数据库 |
| `Planning Agent` 可直接改写计划 | Agent 只能提交 `PlanChangeRequest`，由编排器校验和审批后生效 |
| 质量网关由 Planner 审查 | 拆分为 Schema Validator、Script Validator、LLM Judge、Human Review |
| Worker 加载技能并执行 | Worker 在 Policy Engine 授权下执行，文件、工具、网络、密钥均最小权限 |
| 动态插入步骤 | 通过版本化计划变更协议执行，不直接改主计划 |
| Docker/WASM 作为安全边界 | 增加工具白名单、网络白名单、Artifact 溯源、Secret scoped token、审计日志 |

### 0.3 设计评分

| 维度 | v1 评分 | v2 目标 |
|---|---:|---:|
| 设计目标清晰度 | 7/10 | 9/10 |
| 系统架构完整性 | 6/10 | 9/10 |
| 技术方案落地性 | 5.5/10 | 8.5/10 |

## 1. 背景与痛点分析

### 1.1 多智能体系统的真实落地问题

在企业级 LLM 应用中，多智能体不是“多个 Agent 聊天”这么简单。真实问题是：复杂任务需要多个专业角色协作，但协作过程必须可预测、可恢复、可审计、可干预、可控成本。

当前常见框架存在四类问题。

#### 1.1.1 静态拓扑与黑盒执行

许多框架在任务启动前定义好 Agent 数量、角色和协作路径。一旦任务中出现新情况，例如数据源不可用、需求被重新解释、上游产物质量不足，系统很难安全地调整计划。

用户看到的是一个黑盒流程。Agent 在里面做了什么、为什么这么做、下一步会影响谁，往往不清楚。

#### 1.1.2 上下文膨胀与长期任务失忆

单一 Agent 或长会话容易把全部历史、代码、文档、工具说明塞进上下文。结果是成本上升，注意力下降，幻觉增加。长任务跑到后半程时，Agent 经常忘记最初目标和关键约束。

#### 1.1.3 缺乏确定性的质量控制

很多 Agent 工作流只关注“是否生成了输出”，不关注“输出是否满足契约”。上游产物一旦错误，会沿依赖链向下扩散，最终导致整个交付失败。

#### 1.1.4 人类介入方式低效

完全自主容易失控，每一步都确认又太慢。企业场景需要的是“结构化介入”：在计划锁定、权限放行、异常分支、质量争议等关键点介入，而不是在每个普通动作上打断用户。

## 2. 设计目标

### 2.1 核心目标

1. **声明式驱动**  
   使用 `team.yaml` 描述团队能力，使用 `plan.yaml` 描述任务 DAG、输入输出、验证标准和执行策略。

2. **确定性编排**  
   系统状态由 Orchestration Engine 管理，LLM 不直接写运行状态。

3. **最小上下文执行**  
   每个 Worker 只接收当前 Step 所需的输入 Artifact、工具、权限和少量任务说明。

4. **质量网关强约束**  
   每个 Step 必须经过 Schema、Script、LLM、Human 中一种或多种验证。Required Gate 不通过时不得进入下游。

5. **人机协同可控**  
   用户只在三类节点强制介入：计划锁定、计划变更、高风险失败或权限升级。

6. **可恢复、可审计、可回放**  
   每次 Step 执行、Artifact 生成、计划变更、质量评审、人工审批都写入不可篡改审计日志。

7. **成本可控**  
   每个 Plan、Step、Agent、Validation Gate 都可配置 Token、时间、重试和预算上限。

### 2.2 可衡量指标

| 指标 | 目标 |
|---|---|
| Plan 完整度 | 100% Step 有输入、输出、负责人、验证规则、权限声明 |
| 状态审计 | 100% 状态变更记录 actor、reason、timestamp、trace_id |
| Artifact 溯源 | 100% 产物记录输入依赖、生成 Step、hash、版本 |
| Required Gate | 100% required validation 通过后才允许下游执行 |
| 重试控制 | 单 Step 默认最多 2 次自动重试，超过进入人工处理 |
| 权限最小化 | Worker 默认无网络、无 Secret、只读输入、只写指定输出 |
| 人工介入压缩 | 普通 happy path 不需要每步确认，只在关键点确认 |
| 成本边界 | Plan 和 Step 均可设置预算，超预算自动暂停 |

### 2.3 非目标

v2 明确不做以下事情：

1. 不做完全自主、无审批的长期 Agent。
2. 不允许 LLM 直接修改系统运行状态。
3. 不把 LLM 语义判断作为唯一质量标准。
4. 不支持 Worker 任意访问文件系统、网络或密钥。
5. 不在 MVP 中支持复杂的 Agent 间自由协商协议。
6. 不在 MVP 中支持跨租户共享 Artifact。
7. 不把 `plan.yaml` 当数据库使用。
8. 不优先实现复杂 UI 动画和团队模板市场。

## 3. 核心价值主张

### 3.1 对用户

用户可以在任务执行前看到：谁做、做什么、依赖什么、产出什么、如何验证、失败怎么办。执行中如果系统要改变计划，必须提交变更说明和影响分析。

### 3.2 对工程团队

工程团队获得一个可测试、可审计、可恢复的 Agent Workflow Runtime，而不是一个不可控的聊天式自动化脚本。

### 3.3 对企业资产沉淀

`team.yaml`、`plan.yaml`、验证规则、Agent 模板、Tool Policy、Rubric、Artifact Schema 都可以沉淀为可复用资产。

### 3.4 对成本控制

通过短生命周期 Worker、上下文裁剪、预算上限、小模型执行、大模型规划与审查分层，控制长期任务成本。

## 4. 总体架构

### 4.1 架构分层

v2 将系统拆成五个平面。

```text
+--------------------------------------------------------------------------+
|                              User Interface                              |
|  Team Editor | Plan Timeline | Approval Queue | Execution Monitor        |
+------------------------------------+-------------------------------------+
                                     |
                                     v
+--------------------------------------------------------------------------+
|                         API Gateway / WebSocket                          |
+------------------------------------+-------------------------------------+
                                     |
                                     v
+--------------------------------------------------------------------------+
|                         Control Plane                                    |
|  Team Registry | Plan Registry | Plan Compiler | Scheduler               |
|  State Machine | Policy Engine | Budget Manager | Approval Controller     |
+------------------------------------+-------------------------------------+
                                     |
                                     v
+--------------------------------------------------------------------------+
|                         Execution Data Plane                              |
|  Worker Factory | Sandbox Runtime | Tool Runner | Context Builder         |
+------------------------------------+-------------------------------------+
                                     |
                                     v
+--------------------------------------------------------------------------+
|                         Review Plane                                      |
|  Schema Validator | Script Validator | LLM Judge | Human Reviewer          |
+------------------------------------+-------------------------------------+
                                     |
                                     v
+--------------------------------------------------------------------------+
|                         Storage Plane                                     |
|  Postgres | Object Store | Git | Vector Index | Immutable Audit Log      |
+--------------------------------------------------------------------------+
```

### 4.2 组件职责

| 组件 | 职责 |
|---|---|
| `Team Registry` | 管理团队模板、角色能力、模型策略、工具权限 |
| `Plan Registry` | 管理计划版本、计划锁定、计划变更记录 |
| `Plan Compiler` | 校验 `plan.yaml`，编译成合法 DAG，检查依赖、路径、权限、验证规则 |
| `Scheduler` | 查找 ready Step，发放 lease，控制并发和资源锁 |
| `State Machine` | 维护 Plan、Step、WorkerRun、ValidationRun 的实际状态 |
| `Policy Engine` | 决定 Worker 可读写路径、可用工具、网络、Secret、模型 |
| `Budget Manager` | 控制 Token、时间、调用次数、重试、并发预算 |
| `Approval Controller` | 管理人工审批队列，高风险操作和计划变更必须通过 |
| `Context Builder` | 根据 Step 输入、RAG、必需文件构造最小上下文 |
| `Worker Factory` | 按 Step 定义创建短生命周期 Worker |
| `Tool Runner` | 以受控方式执行工具，记录输入输出和错误 |
| `Quality Gate` | 调用验证器并聚合验证结果 |
| `Artifact Registry` | 管理产物版本、hash、血缘、有效性、存储位置 |
| `Audit Log` | 记录不可篡改的系统事件 |

### 4.3 控制平面与数据平面分离

控制平面只做确定性决策：校验、调度、授权、状态流转、审批。

数据平面只做受控执行：Worker 运行、工具调用、文件读写、产物生成。

LLM 不直接拥有系统状态修改权。它的输出必须进入候选区，由确定性组件处理。

## 5. 核心对象模型

### 5.1 对象关系

```text
TeamTemplate
    |
    +-- AgentRole
            |
            +-- Capability
            +-- ToolPolicy
            +-- ModelPolicy

PlanDefinition
    |
    +-- PlanVersion
            |
            +-- StepDefinition
                    |
                    +-- ArtifactContract
                    +-- ValidationGate
                    +-- ExecutionPolicy
                    +-- PermissionPolicy

PlanRun
    |
    +-- StepRun
            |
            +-- WorkerRun
            +-- ValidationRun
            +-- ArtifactVersion
```

### 5.2 `plan.yaml` 不再保存运行时状态

`plan.yaml` 是期望状态，负责描述“应该怎么做”。

数据库是实际状态，负责记录“现在做到哪里、谁做的、结果如何”。

| 数据 | 存放位置 |
|---|---|
| 任务目标、Step DAG、输入输出、验证规则 | `plan.yaml` |
| 当前运行状态、重试次数、Worker 日志、审批状态 | Postgres |
| Artifact 二进制或大文件 | Object Store |
| 文档、代码、模板、计划定义 | Git |
| 检索索引 | Vector Index |
| 审计事件 | Immutable Audit Log |

### 6.3 `PlanChangeRequest` 规范

动态调整计划时，不允许 Agent 直接修改当前生效计划。必须提交变更请求。

```yaml
change_request:
  id: cr_001
  base_plan_id: liquor_analysis_task_01
  base_plan_version: 1
  proposed_by: planning_agent
  reason: 源数据披露格式不一致，需要增加清洗纠错步骤
  evidence:
    - artifact: artifacts/raw/financial_data.json
      note: 部分企业字段命名不一致，无法直接进入分析步骤
  proposed_changes:
    - operation: insert_step
      after: step_1
      step:
        step_id: step_1_adjust
        title: 财务字段清洗与纠错
        assignee: data_analyst
        depends_on:
          - step_1
  impact:
    affected_steps:
      - step_2
      - step_3
    estimated_time_delta: +20min
    estimated_cost_delta_usd: 2.5
    risk_level: medium
  approval_required: true
  validation_required:
    - schema
    - dag
    - artifact_reference
    - policy
```

## 7. 状态机设计

### 7.1 Plan 状态机

```text
draft
  |
  v
compiled
  |
  v
awaiting_approval
  |
  v
approved
  |
  v
running
  |\
  | \---> paused
  |        |\
  |        | \---> cancelled
  |        v
  |      running
  |
  +---> success
  |
  +---> failed
```

### 7.2 Step 状态机

```text
pending
  |
  v
ready
  |
  v
leased
  |
  v
running
  |
  v
worker_completed
  |
  v
validating
  |\
  | +---> success
  | +---> rework_required ---> pending
  | +---> human_review_required ---> approved ---> success
  |                              \---> rejected ---> rework_required
  |                              \---> force_pass ---> success
  |
  +---> failed

running
  |\
  | +---> timeout
  | +---> cancelled
  | +---> failed
```

### 7.3 Lease 机制

Scheduler 不直接把 Step 标记为 running，而是先发放 lease。

```text
Scheduler
  |
  | find ready steps
  v
Lease Manager
  |
  | acquire step lease with ttl
  v
Worker Factory
  |
  | start worker
  v
StepRun running
```

如果 Worker 崩溃或心跳超时，lease 过期，Step 可重新进入 ready 或 failed，取决于 retry policy。

## 8. 调度与并发模型

### 8.1 DAG 调度规则

Step 可执行条件：

1. 所有 `depends_on` Step 均为 success。
2. 所有 required input_artifacts 存在且有效。
3. Step 未超过最大重试次数。
4. Step 所需资源锁可获得。
5. Plan 未暂停、取消或超预算。
6. Policy Engine 授权通过。

### 8.2 并发控制

```text
Ready Queue
  |
  +-- step_2  requires artifacts/analysis/** lock
  +-- step_4  requires deliverables/** lock
  +-- step_5  requires network:sse.com.cn lock
```

系统需要支持三类锁：

| 锁类型 | 用途 |
|---|---|
| Artifact write lock | 防止多个 Step 写同一路径 |
| External resource lock | 控制同一外部 API 或网站并发 |
| Plan mutation lock | 防止计划变更与调度同时修改依赖图 |

### 8.3 失败恢复

| 场景 | 处理 |
|---|---|
| Worker 超时 | 终止沙箱，记录 timeout，按 retry policy 重试 |
| Worker 部分输出 | 标记 Artifact 为 incomplete，不进入下游 |
| Tool 调用失败 | 记录 tool error，交给 Worker 判断是否可恢复 |
| Validation 失败 | 进入 rework_required 或 human_review_required |
| Plan 动态变更 | 暂停受影响 Step，生成新 plan_version |
| 用户取消 Plan | 终止未完成 Worker，冻结产物，写审计日志 |

## 10. 质量网关设计

### 10.1 验证类型

| 类型 | 作用 | 是否确定性 |
|---|---|---|
| `schema` | 校验结构、字段、类型、必填项 | 是 |
| `script` | 执行业务规则、回归测试、数值校验 | 是 |
| `llm` | 检查语义质量、报告逻辑、自然语言一致性 | 否 |
| `human` | 高风险或主观判断 | 人工 |

### 10.2 分数聚合

```text
ValidationResult
  |
  +-- required gates all passed? no -> block
  |
  +-- weighted score >= threshold? no -> rework
  |
  +-- human checkpoint? yes -> human_review_required
  |
  +-- success
```

### 10.3 Rework 反馈协议

Rework 反馈必须机器可读，不只是一段文本。

```yaml
rework_feedback:
  step_id: step_1
  attempt: 1
  failed_gates:
    - gate_id: completeness_check
      severity: blocking
      reason: 缺少五粮液 2025 年毛利率字段
      evidence:
        artifact: artifacts/raw/financial_data.json
        json_path: $.companies[?(@.name=='五粮液')].gross_margin
  required_fixes:
    - 补齐五粮液 gross_margin 字段
    - 保持已通过校验的字段不变
  forbidden_actions:
    - 不要删除已有企业记录
    - 不要跳过 schema_check
  max_attempts_remaining: 1
```

### 10.4 LLM Judge 约束

LLM Judge 必须遵守：

1. 必须输出结构化评分。
2. 必须引用证据 Artifact 或文本片段。
3. 不得修改验证标准。
4. 不得直接放行 Step。
5. Required Gate 中，LLM Judge 不应作为唯一验证项，除非任务本身是纯自然语言质量判断。

## 11. 计划变更与动态重排

### 11.1 动态变更触发条件

系统允许提出计划变更，但必须走 `PlanChangeRequest`。

触发条件包括：

1. 上游 Artifact 结构与原计划不符。
2. 外部资源不可用，需要替代路径。
3. 验证失败显示原计划缺少必要步骤。
4. 用户修改目标或补充约束。
5. 预算、权限或安全策略阻断原计划。

### 11.2 变更校验流程

```text
Planning Agent proposes change
  |
  v
PlanChangeRequest
  |
  v
Compiler validates schema
  |
  v
DAG checker detects cycles
  |
  v
Policy checker validates permissions
  |
  v
Impact analyzer lists affected steps and artifacts
  |
  v
Human approval if required
  |
  v
New plan_version activated
```

### 11.3 版本策略

| 行为 | 版本处理 |
|---|---|
| 文字说明修正 | patch version |
| Step 验证规则变化 | minor version，需审批 |
| 新增或删除 Step | minor version，需审批 |
| 改变目标或交付物 | major version，需审批 |
| 运行状态变化 | 不改变 plan version |

## 12. 安全模型

### 12.1 威胁模型

| 威胁 | 场景 | 缓解 |
|---|---|---|
| Prompt Injection | 外部网页或文档要求 Agent 泄露密钥 | 外部内容标记为 untrusted data，不作为指令执行 |
| Tool Abuse | Worker 调用未授权工具 | Step-level tool allowlist |
| Path Traversal | Worker 写入计划外路径 | File policy + sandbox mount 限制 |
| Secret Leakage | Worker 读取长期密钥 | scoped token + secret broker + 默认无 Secret |
| Network Exfiltration | Worker 向外部发送敏感数据 | 默认 deny，域名 allowlist |
| Artifact Poisoning | 上游产物污染下游 | Schema、hash、provenance、required validation |
| Plan Tampering | LLM 修改计划绕过验证 | 计划变更必须由编排器编译和审批 |
| Supply Chain | 动态技能或插件不可信 | 技能签名、版本锁定、来源校验 |

### 12.2 最小权限原则

每个 Step 的权限由三部分交集决定：

```text
Role policy from team.yaml
  ∩
Step policy from plan.yaml
  ∩
System policy from admin config
  =
Effective runtime policy
```

### 12.3 Worker 沙箱

Worker 运行时默认配置：

| 权限 | 默认值 |
|---|---|
| 文件系统 | 只挂载输入 Artifact 和输出目录 |
| 网络 | deny |
| Secret | none |
| 进程 | 禁止 fork bomb，限制 CPU 和内存 |
| 时间 | 按 Step timeout 限制 |
| 工具 | allowlist |
| 日志 | stdout、stderr、tool trace 全量记录 |

## 13. Artifact 管理与血缘

### 13.1 Artifact 元数据

每个 Artifact 必须记录：

```yaml
artifact:
  id: raw_financial_data_v3
  logical_name: raw_financial_data
  path: artifacts/raw/financial_data.json
  content_type: application/json
  hash: sha256:abc123
  size_bytes: 123456
  produced_by:
    plan_run_id: run_001
    step_run_id: step_run_001
    worker_run_id: worker_run_001
  inputs:
    - targets/companies.json@sha256:def456
  validation_status: passed
  created_at: 2026-05-22T10:00:00Z
```

### 13.2 Artifact 失效规则

如果上游 Artifact 变更，下游 Step 的输出必须标记为 stale，不能继续复用，除非 Plan 明确声明可复用策略。

```text
targets/companies.json changes
  |
  v
step_1 output stale
  |
  v
step_2 output stale
  |
  v
step_3 output stale
```

## 16. 测试策略

### 16.1 测试金字塔

```text
             E2E Tests
          /-------------\
         Integration Tests
      /---------------------\
     Unit Tests + Schema Tests
  /-----------------------------\
 Static Checks + Policy Tests
```

### 16.2 必测路径

| 模块 | 必测内容 |
|---|---|
| Plan Compiler | Schema 校验、DAG 校验、环检测、Artifact 引用校验 |
| Scheduler | ready Step 选择、lease、并发、资源锁、重试 |
| State Machine | 所有状态转换、非法转换拒绝 |
| Policy Engine | 文件、工具、网络、Secret 权限拒绝和放行 |
| Worker Factory | 上下文注入、沙箱创建、超时终止 |
| Quality Gate | schema/script/llm/human 聚合逻辑 |
| PlanChangeRequest | 变更编译、影响分析、审批、生效 |
| Artifact Registry | hash、版本、stale、血缘 |
| Budget Manager | 超预算暂停、恢复、降级 |

### 16.3 覆盖图

```text
CODE PATH COVERAGE TARGET
=========================
[+] Plan submission
    ├── valid plan -> compiled
    ├── invalid yaml -> rejected
    ├── cyclic dependency -> rejected
    ├── missing artifact reference -> rejected
    └── unauthorized tool -> rejected

[+] Scheduler
    ├── all dependencies success -> ready
    ├── dependency failed -> blocked
    ├── lease acquired -> running
    ├── lease expired -> retry or failed
    └── resource lock conflict -> wait

[+] Worker execution
    ├── success -> worker_completed
    ├── timeout -> timeout
    ├── tool denied -> policy_violation
    ├── partial artifact -> incomplete
    └── crash -> failed

[+] Quality Gate
    ├── all required gates pass -> success
    ├── schema fails -> rework_required
    ├── script fails -> rework_required
    ├── llm score low -> human_review_required or rework
    └── human rejects -> rework_required

[+] Plan Change
    ├── valid change -> new plan_version
    ├── invalid schema -> rejected
    ├── cycle introduced -> rejected
    ├── high risk -> human approval
    └── rejected by human -> no change
```

### 16.4 E2E 场景

1. 用户创建团队，生成 `team.yaml`，审批后入库。
2. 用户提交需求，生成 `plan.yaml`，编译通过，审批后执行。
3. Step 1 成功，Schema 和 Script 验证通过，下游 Step 执行。
4. Step 2 验证失败，生成结构化 Rework 反馈，Worker 重做后通过。
5. Planning Agent 提出插入新 Step，系统生成 `PlanChangeRequest`，用户批准，新版本继续执行。
6. Worker 尝试访问未授权路径，被 Policy Engine 拒绝并记录审计事件。
7. Plan 超预算后暂停，用户批准追加预算后继续。

## 17. 性能与扩展性

### 17.1 潜在瓶颈

| 瓶颈 | 风险 | 方案 |
|---|---|---|
| Planning Agent 频繁唤醒 | 成本高、延迟高 | 仅在计划生成、变更、复杂失败时调用 |
| Artifact 大文件传递 | 上下文膨胀 | 传引用，不传全文，必要时 RAG 摘要 |
| Validation 脚本慢 | 阻塞下游 | 超时、并行验证、缓存验证结果 |
| Postgres 状态写入频繁 | 热点写 | append-only events + 状态快照 |
| Object Store 读取多 | IO 慢 | 本地缓存 + hash 校验 |
| LLM Judge 不稳定 | 评分波动 | Rubric 固定、温度低、抽样复核 |

### 17.2 缓存策略

| 对象 | 缓存条件 |
|---|---|
| Plan 编译结果 | `plan.yaml` hash 不变 |
| Artifact 验证结果 | Artifact hash + validation gate hash 不变 |
| RAG 检索结果 | source files hash 不变 |
| Team capability resolution | team version 不变 |

## 18. MVP 范围

### 18.1 MVP 必须包含

1. 固定团队模板 `team.yaml`。
2. 手写或半自动生成 `plan.yaml`。
3. Plan Compiler，支持 schema、DAG、artifact、policy 校验。
4. Scheduler，支持 DAG ready queue 和 Step lease。
5. 每个 Step 独立 Worker。
6. Artifact Registry 基础版本和 hash。
7. Schema + Script Validation。
8. Rework 机制，最多自动重试 2 次。
9. Human Approval Queue。
10. 基础审计日志。

### 18.2 MVP 暂不包含

1. 全自动团队生成。
2. Worker 间自由协商。
3. 复杂 UI 甘特图编辑器。
4. 多租户 Artifact 共享。
5. 技能市场。
6. 自动动态扩容。
7. 大规模模型路由优化。
8. 跨计划全局优化。

## 19. 技术选型建议

### 19.1 推荐栈

| 层 | 建议 |
|---|---|
| API | FastAPI 或 NestJS |
| 状态库 | Postgres |
| 队列 | Redis Queue、Temporal、Celery、BullMQ 之一 |
| 长工作流 | 优先评估 Temporal，避免手写复杂重试和状态恢复 |
| Object Store | S3 兼容存储或 MinIO |
| 沙箱 | Docker MVP，后续评估 Firecracker 或 WASM |
| Schema | JSON Schema + Pydantic/Zod |
| DAG | networkx 或自实现 Kahn 算法 |
| 前端 | React + Timeline/Approval Queue |
| 审计 | append-only event table，后续接入不可篡改日志存储 |

### 19.2 Boring by Default 建议

不要一开始自研完整工作流引擎。调度、重试、超时、补偿、恢复，这些问题非常容易低估。

优先路线：

1. MVP 可以用 Postgres + 简单队列实现。
2. 一旦出现跨小时任务、复杂重试、恢复和补偿，优先评估 Temporal。
3. 不要把所有编排逻辑交给 LLM。

## 20. 风险分析与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 无限 Rework | Worker 多次失败 | max_attempts + human escalation |
| Planner 幻觉 | 生成非法计划 | Plan Compiler + Schema + DAG Check |
| 状态竞争 | 多 Worker 写同一状态 | lease + resource lock + DB transaction |
| 计划漂移 | 动态变更偏离目标 | PlanChangeRequest + impact analysis + approval |
| 成本失控 | 重试和 LLM Judge 过多 | Budget Manager |
| Prompt Injection | 外部内容诱导 Agent 越权 | untrusted content boundary + policy engine |
| Artifact 污染 | 错误产物传到下游 | validation gate + artifact provenance |
| RAG 漏召回 | Worker 缺少关键上下文 | required files + retrieval evidence |
| 人工审批拥塞 | 多计划卡在审批 | Approval Queue + priority + SLA |
| 审计不足 | 失败后无法追责 | immutable audit events |

## 21. NOT in scope

以下内容经过评审，建议明确延后。

| 不在范围 | 原因 |
|---|---|
| 完全自主动态组织架构 | 风险高，难以验证，先用固定团队模板 |
| Agent 间自由多轮协商 | 可观测性差，容易失控，MVP 先走中心调度 |
| 复杂可视化拖拽编排器 | 不是核心风险，先用 YAML + 简单审批 UI |
| 跨租户模板市场 | 涉及权限、隐私、供应链风险 |
| 自动动态扩容 | 需要成熟预算和资源模型后再做 |
| LLM 直接修改生产计划 | 破坏可控性，必须走 Change Request |
| LLM 作为唯一质量门禁 | 不可重复，必须有确定性验证兜底 |

## 23. Failure Modes

| 新路径 | 生产失败方式 | 是否有测试建议 | 是否有错误处理 | 用户感知 |
|---|---|---|---|---|
| Plan 编译 | LLM 生成环形依赖 | DAG 单元测试 | 编译拒绝 | 用户看到计划非法原因 |
| Step 调度 | 两个 Worker 抢同一 Step | Lease 并发测试 | DB lease | 用户无感 |
| Artifact 写入 | Worker 输出半成品 | Artifact completeness 测试 | incomplete 状态 | 用户看到 Step 失败原因 |
| Validation | Script 超时 | Timeout 测试 | validation timeout | 用户看到验证超时 |
| Rework | 同一错误重复出现 | max retry 测试 | human escalation | 用户收到人工介入请求 |
| Plan Change | 新 Step 造成依赖环 | ChangeRequest DAG 测试 | 变更拒绝 | 用户看到变更未通过 |
| Policy | Worker 访问未授权网络 | Policy deny 测试 | 阻断并审计 | 用户看到权限违规 |
| Budget | Token 超限 | Budget 测试 | Plan paused | 用户看到追加预算选项 |
| RAG | 漏召回关键文件 | Retrieval eval | required files 强制注入 | 用户看到上下文不足提示 |

没有发现必须在 MVP 外静默接受的关键失败。所有关键失败都应被测试或显式错误处理覆盖。

## 24. 并行实施策略

| 工作流 | 模块 | 依赖 |
|---|---|---|
| Schema 与 Compiler | schemas、compiler | 无 |
| State Machine 与 DB | state、models、migrations | Schema 基础定义 |
| Scheduler 与 Lease | scheduler、queue | State Machine |
| Policy Engine | policy、runtime | Schema 基础定义 |
| Worker Runtime | runtime、tools | Policy Engine |
| Quality Gate | validation、artifacts | Artifact Registry |
| Artifact Registry | artifacts、storage | State Machine |
| Approval Queue | approval、api、ui | State Machine |
| Observability | audit、metrics | State Machine |

### 24.2 并行 lanes

```text
Lane A: Schema + Plan Compiler
Lane B: State Machine + DB models
Lane C: Policy Engine

After A+B:
Lane D: Scheduler + Lease
Lane E: Artifact Registry

After C+D:
Lane F: Worker Runtime

After E:
Lane G: Quality Gate

After B:
Lane H: Approval Queue + Audit UI
```

### 24.3 执行顺序

1. 并行启动 Lane A、B、C。
2. 合并后启动 Lane D、E、H。
3. Lane C 和 D 完成后启动 Lane F。
4. Lane E 完成后启动 Lane G。
5. 最后做 E2E 串联测试。

## 26. v2 结论

DDMAO 的核心机会不是“多创建几个 Agent”，而是把多 Agent 工作流变成可声明、可编译、可调度、可验证、可审计的工程系统。

v2 相比 v1 的关键提升是：

1. 把 YAML 从“状态存储”降级为“期望状态声明”。
2. 把 Planner 从“总监理 + 执行权”降级为“建议生成者”。
3. 把 Quality Gate 从“LLM 评审”升级为“确定性验证优先，LLM 语义兜底”。
4. 把动态改计划升级为版本化 Change Request。
5. 把沙箱安全升级为完整 Policy Engine。
6. 把执行流程升级为有 lease、锁、预算、审计、Artifact 血缘的生产级工作流。

推荐下一步不是继续扩写概念，而是实现 MVP：

```text
team.yaml + plan.yaml
        |
        v
Plan Compiler
        |
        v
Scheduler + StepRun DB
        |
        v
Sandbox Worker
        |
        v
Artifact Registry
        |
        v
Schema + Script Quality Gate
        |
        v
Human Approval on failure
```

这条路径足够小，也足够完整。先把这条链跑通，再讨论动态扩容、多团队模板、复杂 UI 和模型路由。
