## 1. Summary

**多智能体团队系统（Multi-Agent Team System，下称 MATS）让用户用一句"团队目标"动态组建一支 AI 团队，由一个 Leader 智能体统一指挥若干专业智能体，完成从任务分解、分配、执行到结果汇总的闭环；用户可在执行过程中随时插话调整，由 Leader 决定中断、改派或重做。**

MATS 不是"多个 Agent 各自聊天"，而是一个**中心化、可干预、可观测**的团队协作运行时。它基于 Claude Agent SDK（Python）构建，复用 goktech-agent 平台已有的 skills、plugins、command 资产，把"找不到的能力用 prompt 现场补齐"，从而支撑 FDE 小队在客户现场快速拉起一支可用的 AI 团队。

本 PRD 聚焦四个核心能力：**动态组建团队、Leader 任务分解与分配、运行中人工干预、Leader 中心化协调**。

## 3. Background

### 3.1 当前背景

goktech-agent 平台已基于 Claude Agent SDK 跑通单 Agent 能力，并沉淀了一批 skills、plugins 和 command。但目前面对"需要多个角色协作才能完成"的真实交付任务（如一份完整的售前方案、一次品类调研到运营的全链路、一份带调研+撰写+审核的报告），仍然依赖人工在多个单 Agent 会话之间手动搬运上下文。

平台此前已有两份相关沉淀：

- 《多 Agent 协作平台技术方案讨论稿》——确认了 Leader 中转 + 自建 Mailbox 的方向，并指出 Claude Agent SDK 不原生支持 Agent Teams/Mailbox，需自建。
- 《DDMAO v2 设计文档》——一套声明式（`team.yaml` + `plan.yaml`）、强调确定性编排与质量网关的重型企业级方案。

MATS 是一套**独立的新设计**：相比 DDMAO v2 的"声明式 + 重编排"，MATS 走"**对话式动态组队 + Leader 智能调度 + 人在环干预**"的轻量路线，目标是先在 AI 实训营和 FDE MVP 中快速可用。两者面向不同成熟度阶段，可在后续融合（MATS 产出的团队可导出为 DDMAO 的 `team.yaml` 资产）。

### 3.2 为什么现在建设

- AI 实训营和首批 FDE 试点都需要"开箱即用的团队协作"，而不是让用户自己拼 Agent。
- 平台已积累足够的 skills/plugins，"动态组队时自动复用已有能力"具备落地条件。
- 用户的真实工作流是**边做边调**——目标会变、中途会插需求，需要一个能听懂"停一下、改一下、重做一下"的协调者。

### 3.3 产品机会

MATS 可以成为 goktech-agent 平台从"单 Agent 工具"迈向"AI 团队工作台"的关键一跳。它把"组队—分工—执行—干预—汇总"做成一个可复用的运行时，既服务 FDE 的 0→1，也为 AI CoE 沉淀"团队模板 + 角色模板"资产。

## 4. Objective

### 4.1 产品目标

让用户用自然语言描述一个团队目标，系统就能在澄清几个关键问题后，自动组建一支角色清晰、技能明确的 AI 团队，并由 Leader 指挥团队完成一个具体任务；整个过程用户可随时干预，结果可汇总、可沉淀。

### 4.2 业务目标

- 把"多角色协作任务"的人工搬运成本降到接近零。
- 最大化复用平台已有 skills/plugins，减少重复造轮子。
- 让非技术用户也能"指挥一支 AI 团队"，而不需要懂提示词工程。
- 把一次成功的组队沉淀为可复用的团队模板。

### 4.3 Key Results

| 编号 | Key Result | 衡量方式 |
| --- | --- | --- |
| KR1 | 支持从一句团队目标动态组建团队 | 用户输入目标 → 系统澄清 → 确认 → 自动生成 ≥3 个子智能体 |
| KR2 | 组队时能自动检索并复用已有 skills/plugins | 角色技能中能正确命中系统已存在的 skill/plugin，未命中的用 prompt 兜底 |
| KR3 | Leader 能自动分解任务并形成任务列表 | 给定一个团队问题，Leader 产出带依赖关系的任务列表并分配到角色 |
| KR4 | Leader 能按依赖调度执行并汇总结果 | 任务按 串行/并行/讨论 模式执行，Leader 汇总成一份交付物 |
| KR5 | 运行中支持用户插话干预 | 用户中途输入新要求，Leader 能中断/改派/增删改任务/令某 Agent 重做 |
| KR6 | 全流程经 Leader 中心化协调 | Agent 间不直接通信，讨论与协调消息均经 Leader 中转并留痕 |
| KR7 | 团队与运行过程可观测、可导出 | 任务列表、Agent 状态、消息往来实时可见；团队可导出为模板 |

## 5. 用户与场景

### 5.1 主要用户

| 用户 | 核心任务 | 痛点 |
| --- | --- | --- |
| FDE Delta | 现场快速拉起一支团队做方案/Demo | 手动拼 Agent、搬上下文太慢，能力难复用 |
| FDE Echo | 用团队完成调研+整理+复盘 | 多步骤任务难以一次性交给"一个会分工的助手" |
| AI CoE | 沉淀团队模板与角色模板 | 缺少把"一次成功组队"标准化复用的载体 |
| 业务/试点用户 | 给一个目标，看团队产出结果 | 不懂提示词，希望"说人话就能指挥团队" |

### 5.2 典型场景

**场景 A：从目标到团队（动态组队）**
用户输入"我要一支帮我做电网 AI 售前方案的团队" → 系统反问 3 个关键问题（交付物形式？客户行业细分？是否需要竞品分析？）→ 给出建议角色（需求分析师、方案撰写、竞品调研、评审）与各自技能 → 用户确认 → 系统检索到 `presales-req-analysis`、`competitor-analysis` 等已有 skill 直接绑定，未命中的"方案评审"角色用 prompt 现场构造 → 团队就绪。

**场景 B：从问题到交付（Leader 调度）**
用户给团队一个问题"基于这份招标文件输出一版应答方案" → Leader（planning 角色）分解为：解析招标文件 → 提取评分项 → 分头撰写各章节 → 汇总评审 → 形成任务列表并分配 → 各 Agent 执行 → Leader 汇总成一份方案。

**场景 C：运行中插话（人工干预）**
执行到"撰写章节"时用户插话"竞品部分要重点对比 XX 厂商" → Leader 判断：暂停撰写 Agent，向竞品调研 Agent 新增一个子任务，完成后让撰写 Agent 重做受影响章节，其余任务不受影响继续。

## 6. Value Proposition

- **对用户**：说一句话就有一支会分工的 AI 团队；中途能像指挥真人一样喊停、改派、重做。
- **对 FDE**：现场组队从"小时级手动拼装"降到"分钟级对话生成"，且自动复用平台已有能力。
- **对 AI CoE**：每一次成功组队都能沉淀为团队模板与角色模板，形成可复用资产库。
- **对平台**：让已有 skills/plugins 资产被真正"组合调用"，提升资产复用率与可见的业务价值。

## 7. Solution

### 7.1 产品结构

MATS 在 goktech-agent 平台内以「团队工作台」形态存在，包含四个区域：

| 区域 | 作用 | 关键能力 |
| --- | --- | --- |
| 组队向导 | 把目标变成团队 | 目标输入、澄清问答、角色/技能建议、技能匹配、确认组队 |
| 团队总览 | 看团队构成 | 角色卡片、各自技能/工具、来源（已有 skill / plugin / prompt 兜底） |
| 任务看板 | 看 Leader 怎么分工 | 任务列表、依赖关系、负责人、状态（待办/进行/完成/重做/阻塞） |
| 协作时间线 | 看团队怎么干活 | Leader↔Agent 消息、讨论记录、人工干预记录、最终汇总 |

### 7.2 核心用户流程

#### 流程一：动态组建团队

```
用户输入团队目标
   │
   ▼
组队向导：分析目标 → 生成澄清问题（角色边界/交付物/约束）
   │
   ▼
用户回答澄清问题
   │
   ▼
系统产出"团队蓝图"建议：
   • 角色列表（含 Leader）
   • 每个角色应具备的技能
   • 每个角色的产出契约
   │
   ▼
技能匹配引擎：
   • 在 skills 库中检索匹配的 SKILL
   • 在 plugins 库中检索匹配的 plugin/command
   • 命中 → 直接绑定
   • 未命中 → 用 prompt 现场构造该角色的能力
   │
   ▼
用户确认/微调团队蓝图
   │
   ▼
实例化团队：创建 Leader + N 个子智能体
   │
   ▼
团队就绪（可保存为团队模板）
```

#### 流程二：Leader 分解任务并执行

```
用户向团队提出一个待解决的问题
   │
   ▼
Leader（planning 能力）理解问题 → 任务分解 → 形成任务列表（带依赖）
   │
   ▼
Leader 把任务分配给对应角色，并决定执行模式：
   • 串行（有依赖）
   • 并行（无依赖）
   • 讨论（需要多角色协作/交叉验证，经 Leader 中转）
   │
   ▼
子智能体执行各自任务，产出结果 → 回报 Leader
   │
   ▼
Leader 校验/串联结果（必要时触发重做）
   │
   ▼
Leader 汇总成最终交付物 → 返回用户
```

#### 流程三：运行中人工干预

```
执行过程中，用户输入新要求 / 新约束 / 纠偏指令
   │
   ▼
Leader 评估影响：
   • 是否需要中断某个正在执行的 Agent
   • 任务列表如何增 / 删 / 改 / 查
   • 哪些任务需要重做、哪些不受影响
   │
   ▼
Leader 执行调整：
   • interrupt 正在跑的 Agent
   • 更新任务列表与依赖
   • 重新分配 / 重做受影响任务
   │
   ▼
向用户说明调整方案与影响范围 → 继续执行
```

### 7.3 Key Features

#### 7.3.1 动态组建团队（需求 1）

| 功能 | 描述 | 首版范围 |
| --- | --- | --- |
| 团队目标输入 | 用户用自然语言描述团队目标 | 必须 |
| 澄清问答 | 系统就角色边界、交付物、约束生成并询问关键问题 | 必须 |
| 团队蓝图建议 | 产出角色列表、各角色技能、产出契约 | 必须 |
| 技能匹配引擎 | 检索已有 skills / plugins / command 并绑定 | 必须 |
| Prompt 兜底 | 未命中的能力用 system prompt 现场构造角色 | 必须 |
| 用户确认与微调 | 增删角色、调整技能、改名 | 必须 |
| 团队模板沉淀 | 组队结果可保存/复用为模板 | 首版基础支持 |

#### 7.3.2 Leader 任务分解与分配（需求 2）

| 功能 | 描述 | 首版范围 |
| --- | --- | --- |
| 默认 Leader | 默认配置一个具备 planning 能力的 Leader Agent | 必须 |
| 任务分解 | 把团队问题拆成带依赖的任务列表 | 必须 |
| 任务分配 | 按角色能力把任务指派给子智能体 | 必须 |
| 执行模式 | 支持串行 / 并行 / 讨论三种调度 | 必须（讨论模式首版基础） |
| 结果汇总 | Leader 串联各 Agent 产出为最终交付物 | 必须 |

#### 7.3.3 运行中人工干预（需求 3）

| 功能 | 描述 | 首版范围 |
| --- | --- | --- |
| 插话输入 | 执行中用户可随时输入新要求 | 必须 |
| 中断判断 | Leader 判断是否中断某个 Agent 的执行 | 必须 |
| 任务增删改查 | Leader 对任务列表做 CRUD | 必须 |
| 重做指派 | 让某个 Agent 重做某个任务 | 必须 |
| 影响说明 | 调整前向用户说明影响范围 | 必须 |

#### 7.3.4 Leader 中心化协调（需求 4）

| 功能 | 描述 | 首版范围 |
| --- | --- | --- |
| Leader 中转通信 | Agent 间不直接通信，全部经 Leader 转发 | 必须 |
| 讨论协调 | 多角色讨论由 Leader 主持与传递 | 必须 |
| 全程留痕 | 所有协调/讨论/干预消息可观测、可审计 | 必须 |

### 7.4 与已有方案的关系

| 维度 | MATS（本系统） | DDMAO v2 |
| --- | --- | --- |
| 组队方式 | 对话式动态组队（LLM 建议 + 用户确认） | 声明式 `team.yaml` 模板 |
| 调度 | Leader 智能调度（LLM 决策为主） | 确定性编排器（DAG + 编译器） |
| 干预 | 运行中自然语言插话 | 版本化 PlanChangeRequest + 审批 |
| 质量门控 | Leader 校验 + 可选脚本/人工 | 强制 Schema/Script/LLM/Human 网关 |
| 定位 | 轻量、快速可用、实训营/MVP 首选 | 重型、企业级、生产可控 |
| 关系 | 独立设计；可把团队导出为 v2 `team.yaml` 资产 | MATS 的上层演进目标 |

### 7.5 Assumptions

| 假设 | 风险 | 验证 |
| --- | --- | --- |
| Claude Agent SDK（Python）能支撑持久会话 + 中断 + 自建中转 | 若 SDK 能力不足，需降级为 subagent 模式 | 用最小原型验证 ClaudeSDKClient 的 interrupt 与多会话 |
| 平台 skills/plugins 元数据足够用于自动匹配 | 元数据稀疏会导致匹配率低 | 用真实 skill 库跑匹配召回测试 |
| 用户愿意先回答几个澄清问题再组队 | 问题太多会劝退 | 澄清问题控制在 3–5 个 |
| Leader 中心化不会成为严重瓶颈 | 大团队下 Leader 负载高、延迟大 | 首版限制团队规模 ≤ 6 |
| Token 成本可接受 | 多 Agent 并行 ≈ 多倍消耗 | 按需唤醒、子任务用较小模型 |

## 8. Release

### 8.1 版本策略

先跑通"组队 → 分解 → 执行 → 汇总"的单团队闭环，再补"运行中干预"，最后做"模板沉淀与可观测"。

### 8.2 V0.1：单团队闭环（最小可用）

目标：一句目标组出团队，Leader 分解并执行，产出一份交付物。

范围：
- 团队目标输入 + 澄清问答（固定 3–5 题）。
- 团队蓝图建议 + skills/plugins 基础匹配 + prompt 兜底。
- 默认 planning Leader + 任务分解 + 串行/并行执行 + 结果汇总。
- Leader 中转通信，全程消息留痕。
- 任务看板与协作时间线的基础展示。

不包含：讨论模式的复杂多轮协商、团队模板市场、细粒度权限。

### 8.3 V0.2：人工干预与讨论

目标：支撑"边做边调"。

范围：
- 运行中插话 → Leader 中断/改派/任务 CRUD/重做。
- 讨论模式（多角色经 Leader 协作与交叉验证）。
- 干预影响分析与说明。
- 团队/角色模板沉淀与复用。

### 8.4 V1.0：可观测与资产化

目标：可统计、可复用、可沉淀。

范围：
- 完整运行可观测（状态、消息、token、成功率）。
- 团队导出为 DDMAO `team.yaml` 资产。
- 与平台权限/空间隔离打通。
- 定时/触发式团队任务（轻量）。

## 9. MVP 验收标准

| 验收项 | 标准 |
| --- | --- |
| 动态组队 | 用户输入一句目标，经澄清后系统生成 ≥3 个角色及其技能，用户可确认 |
| 技能复用 | 角色技能能正确命中系统已有 skill/plugin；未命中项有 prompt 兜底 |
| 任务分解 | Leader 对一个团队问题产出带依赖关系的任务列表并分配到角色 |
| 任务执行 | 任务能按串行/并行执行，子 Agent 产出结果并回报 Leader |
| 结果汇总 | Leader 把各 Agent 结果汇总为一份完整交付物 |
| 人工干预 | 用户中途插话，Leader 能中断/改派/增删改任务/令某 Agent 重做并说明影响 |
| 中心协调 | Agent 间不直接通信，协调与讨论均经 Leader 中转且全程留痕 |
| 可观测 | 任务列表、Agent 状态、消息往来实时可见 |

## 10. Open Questions

1. 团队规模上限首版定为多少（建议 ≤6）？是否需要支持 Leader 之下再设子 Leader？
2. 澄清问答是固定题库，还是由 LLM 动态生成？两者可否结合（动态生成 + 兜底题库）？
3. 技能匹配的召回口径：仅按 SKILL `description` 语义匹配，还是引入标签/分类？
4. "讨论模式"首版做到什么程度——是真多轮协商，还是 Leader 串行征询各角色意见后汇总？
5. 子智能体默认模型策略（Leader 用 Opus、Worker 用 Sonnet/Haiku）由谁定、能否用户覆盖？
6. 干预导致的"重做"是否需要保留旧产物版本以便对比/回滚？
7. 团队/角色模板的存储与权限是否复用平台项目资产库？
