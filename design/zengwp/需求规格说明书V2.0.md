## 一、产品定义

### 1.1 产品名

**AI 顾问作战台**

### 1.2 与现有系统的关系

本应用复用智能体平台底座（`agent-platform-main`）的以下基础设施：

| 复用项 | 说明 |
|--------|------|
| `User` 模型 + Session 认证 | ~~复用企微认证~~ → **重构为自建注册登录体系**（手机号/用户名+密码），企微认证代码注释保留以备未来扩展 |
| `Agent` → `Session` 对话骨架 | 扩展为"项目级会话" |
| `wrapper.py` LLM 调用封装 | 含 thinking 模式、结构化输出 |
| `web_research/` 搜索管线 | Tavily/Firecrawl/Bing |
| SSE 流式对话 | Chat Stream 实时推送 |
| 个人空间/团队空间文件系统 | 知识资产文件归档 |
| 数据库 + 迁移机制 | PostgreSQL + SQLAlchemy 2.0 |

**新增部分**：Customer、Project、ProjectDepartmentAccess、Organization、StakeholderRelation、BusinessMapVersion、BusinessMapDraft 等业务实体，以及对应 API 和前端页面。

### 1.3 一句话定位

一个让轻咨询部用户做"假设地图 → 拜访方案 → 现场验证 → 现状地图 → 营销地图 → 知识沉淀"全流程统一作业的 AI 平台。

### 1.4 使用范围

- 第一阶段供轻咨询部内部测试
- 1-3 名用户
- 1-2 个种子项目

### 1.5 最小闭环

```
用户注册/登录 → 创建客户 → 创建项目 → 对话中上传资料
      ↓
生成假设地图 L1→L2→L3→L4（基于公开信息搜索）
      ↓
生成拜访前方案 → 去客户现场
      ↓
回填拜访纪要 → 验证假设 → 更新现状
      ↓
生成/更新营销地图（角色卡）
      ↓
复盘 → 知识沉淀 → 进入下一轮
```

### 1.6 核心原则

1. **对话是唯一的 AI 交互入口** — 所有工作流在对话中触发、产出、确认；对话页不承担数据深度编辑
2. **业务地图 / 营销地图 / 拜访记录为独立数据页面** — 各自顶部带项目选择器，负责展示、增删改查、深度编辑；不承担 AI 交互
3. **所有 WF 产出支持 Chat 调整** — 产出 → 对话修改 → 再产出 → 直到采纳或驳回
4. **假设地图永久存档** — 现状地图独立新建并关联回假设，复盘期可对比
5. **项目内全透明，项目外全隔离** — 被邀请进项目即可见全部数据，跨项目默认不可见

## 二、导航与页面设计

### 2.1 总体导航结构

业务地图、营销地图、拜访记录作为**独立一级页面**，与对话平级。各自顶部带项目选择器来切换数据上下文。

```
Sidebar
├── 💬 对话                   ← AI 交互入口（选择客户/项目后发起会话）
├── 🗺️ 业务地图               ← 独立页面（顶部选项目，展示 + 编辑 L1-L4 地图）
├── 🎯 营销地图               ← 独立页面（顶部选项目，展示 + 编辑角色卡五视图）
├── 📝 拜访记录               ← 独立页面（顶部选项目，展示 + 筛选证据时间线）
├── ─────────────────────
├── 📁 个人空间               ← 文件存储，项目筛选框 + 自动归档
├── 👥 团队空间               ← 公开资产 + 方法论库
├── 🤖 智能体管理              ← 现有功能保留
└── ⚙️ 系统设置（admin 可见，4-tab 统一页面）  ← 🆕 V2.4
    ├── 用户管理              ← 左侧组织架构树 + 右侧用户列表（角色分配/状态/审批）
    ├── 角色管理              ← 角色 CRUD + 菜单权限勾选
    ├── 菜单管理              ← 菜单树配置（驱动侧边栏动态渲染）
    └── 日志管理              ← 审计日志查询（筛选+表格+展开详情）
```

**为什么不用 Tab 而用独立页面？**

| 对比维度 | Tab 方案（已废弃） | 独立页面方案（✅ 采用） |
|----------|-------------------|------------------------|
| 数据归属 | 一个页面下多个 Tab，不同 Session 产出的数据"自动合并"到一个地图，合并规则不清晰 | 每个页面顶部选项目，数据源 = 该项目全部已确认的正式数据，单一真源 |
| 权限控制 | 进项目就看到所有 Tab，隐式全开放 | 页面级权限独立，未来可按需扩展（如"只读角色只能看不能编辑地图"） |
| 用户心智 | "我在项目里做所有事情" | "我要看地图 → 点地图；我要聊天 → 点对话" |
| 可扩展性 | 项目下新增功能只能加 Tab | 新增功能独立成页，Sidebar 不受影响 |

### 2.2 页面 1：💬 对话

**定位**：唯一 AI 交互入口。所有工作流的触发、产出、候选区审核在此完成。

**布局**：复用现有 Chat 页面的三栏结构。

| 区域 | 内容 |
|------|------|
| **顶部栏** | 项目选择器（客户/项目下拉切换）+ 项目阶段标签 + 项目成员 |
| **左侧会话列表** | **按当前选中项目过滤**的历史会话（按时间倒序）+ 新建对话按钮。选项目N 只显示该项目会话；不选项目显示全部（含未绑定项目的自由对话会话）。切换项目时列表实时刷新 |
| **中间消息流** | ChatStream：用户消息 + AI 回复（含候选结果卡 + 可展开思考过程 + 来源证据） |
| **中间 Banner** | WF07 待审核提醒（Owner 可见）、角色消歧确认 |
| **底部输入区** | 工作流 chips + 文本输入 + 文件拖入区 + 发送按钮 |

> 🆕 V2.5 项目隔离规则（详见 §2.2.1）：选中项目后对话列表实时按项目过滤；切换项目时若当前打开的会话不属于新选项目则中断流式输出并清空右侧、回到空状态；当前选中项目持久化到 localStorage，刷新不丢，失效静默降级回自由对话模式。

**工作流 chips**（5 个）：

| chip | 对应 Skill | 说明 |
|------|-----------|------|
| 📋 生成假设地图 | WF07 | 分步生成 L1→L2→L3→L4 |
| 📝 生成拜访前方案 | WF06 | 无前置依赖，随时可用 |
| 🎙️ 整理拜访纪要 | WF09 | 上传文件或粘贴文本 |
| ✅ 验证假设 & 更新现状 | WF10 | 对比假设 vs 证据 |
| 🎯 营销地图（角色卡） | WF12 | 生成/更新角色卡 |

**文件上传**：拖入文件 → 自动触发 Skill WF02（分类 + 归档到个人空间对应项目目录）。用户确认分类后完成归档。

**交互闭环**：

```
用户输入 / 装填 WF chip
      ↓
AI 执行 Skill → 产出候选结果（Markdown + 结构化数据预览）
      ↓
展示"可展开思考过程"（DeepSeek reasoning_content）
      ↓
用户不满意 → Chat 自然语言调整 → AI 重新生成 → 展示 diff
      ↓
用户采纳 → 结构化数据进正式库 → 对应数据页面可见
用户驳回 → 进偏差池/驳回池
```

**Chat Mode**：不装填 WF chip 时自由对话。提供"标记为有价值"按钮 → 内容自动存为个人空间对应项目目录下的 Markdown 知识片段。

#### 2.2.1 项目隔离对话规则（🆕 V2.5 新增）

基于 2026-07-15 需求讨论。解决"选了项目1却仍看到全部项目的对话混在一起"问题。决策记录见 DECISIONS.md #72–#77。

**核心规则**：

| 规则 | 说明 |
|------|------|
| **严格过滤** | 选中项目N → 左侧会话列表**只显示 `project_id=项目N` 的会话**；选中项目2 → 只显示项目2会话。未绑定项目的自由对话会话（`project_id=null`）在选中项目时被排除 |
| **不选项目=自由对话模式** | 不选项目 → 列表显示全部（含未绑定的自由对话会话），用户可在对话框自由与大模型 chat 问答生成，但**无法触发业务地图/营销地图联动**（项目联动由会话 `project_id` 绑定决定，自由对话会话无项目绑定则不挂载草稿工具等联动逻辑） |
| **切换即切换** | 在项目1的会话 X 打开时切到项目2 → 列表实时变更为只显示项目2会话；若 X 不属于项目2（含 X 是未绑定的自由对话会话）→ **中断 X 在跑的流式输出（调 stopSession）+ 清空右侧消息区 + currentId 置空**，回到空状态提示"选择项目2的会话或新建"；若 X 恰好属于项目2 → 保持打开，内容不丢 |
| **新建会话归属跟随选择** | 选中项目时新建 → 自动绑 `project_id=该项目`；不选时新建 → `project_id=null` 自由对话会话 |
| **与工作区维度叠加** | 项目过滤与现有「个人/团队/全部」工作区维度 **AND 叠加**，不破坏现有切换：在"全部"模式下选项目1 显示可访问的所有项目1会话（个人+团队），在"个人"模式下选项目1 显示个人项目1会话 |
| **团队空间同样适用** | 团队空间会话也带 `project_id`，选中项目时团队空间内会话同样按项目过滤 |
| **当前项目持久化** | 选中项目后写入 localStorage（仅存 id）；刷新页面后用该 id 从 `listProjects()` 重拉完整 Project 对象（含 my_role 等动态字段）恢复为当前选中，会话列表自动按该项目过滤 |
| **失效静默降级** | localStorage 存的 id 已失效（项目被删/用户被移出该/权限变更）→ 清除该 key + 回退到自由对话全量模式，不弹提示、不报错、不打断用户 |
| **后端可见性兜底** | 后端按 `project_id` 过滤时**不额外查项目权限**，复用现有 `workspace_kind`（personal 仅本人）/`mine_only`/TeamSpaceMember 可见性闭环：即使传了已失效的 project_id，用户也只能看到本就有权看到的会话，不越权泄露 |

**实施范围**：仅对话页（ChatWorkspace）与团队空间对话列表。业务地图/营销地图/拜访记录三页已按 `selectedProject` 筛选，不动；知识库（团队共享方法论，非项目作用域）、智能体/技能/用量分析/反馈/登录白名单/系统设置等全局或管理性页面不按项目筛选。

**数据前提**：无需新增模型。`ChatSession.project_id`（§5.2 ChatSession）已存在，`SessionOut` 已返回 `project_id`/`project_name`。本规则仅补齐「列表过滤参数 + 前端切换联动 + 持久化」三处缺口。

### 2.3 页面 2：🗺️ 业务地图

**定位**：独立数据页面。假设地图 + 现状地图 + 偏差池的展示与深度编辑。AI 在对话中产出候选 → 用户采纳后 → 数据正式写入 → 本页面可见和可编辑。

**数据来源**：该项目下所有 Chat Session 产出的、已确认（reviewStatus = reviewed）的 BusinessMapObject。单一数据源，不做多 Session 合并。

**布局**：

**子视图**：

- **假设地图**（默认）：展示 mapType=hypothesis 的 L1-L4 节点树
- **现状地图**：展示 mapType=current 的节点，结构相同。节点多出 `verificationStatus` 标记 + `linkedHypothesisId`（可点击跳回原假设节点）
- **偏差池**：展示所有 `verificationStatus = 推翻` 的节点，假设 vs 现状逐条对比，红色高亮
- **前置分析**（🆕）：产业环境与战略定位分析（5维度：行业价值链、客户行业地位、行业趋势与变化、客户战略定位、数字化驱动力），回答"为什么客户当前的战略是这样？数字化为什么是其关键抓手？"。数据存储为项目级 PreAnalysis 对象
- **五维健康**（🆕）：L1公司级五维健康总览（雷达图+各价值链环节对比表）+ L2/L3级逐域/逐场景健康评分。点击表行可跳转查看对应节点详情

**五维健康观测体系**（贯穿 L1-L3）：

| 维度 | 在L1的含义 | 在L2的含义 | 在L3的含义 |
|------|-----------|-----------|-----------|
| L5 数字意识 | 战略是否分解到各价值链环节？IT投资与业务战略是否匹配？ | 域目标是否量化、是否分解至具体岗位KPI？ | 场景目标是否支撑域目标？岗位是否理解战略意义？ |
| L4 数字神经 | 跨环节流程是否顺畅？协同效率如何？变更是否可控？ | 域内价值流流转效率、跨岗位协同效果 | 场景内步骤衔接效率、异常响应速度 |
| L3 数字器官 | 核心IT系统是否覆盖各环节？系统间是否集成？用户是否愿意用？ | 支撑IT系统功能覆盖度、用户采纳率、NPS | 使用的系统功能是否好用？用户满意度 |
| L2 数字血液 | 跨价值链关键数据是否准确、及时、共享？ | 关键数据实体的质量、时效性、共享程度 | 场景内数据流的准确性、实时性、完整性 |
| L1 数字骨架 | 基础设施是否弹性、经济、可持续？ | 域内基础设施资源消耗效率、SLA满足度 | 场景执行所需资源的弹性、成本、可用性 |

**定位**：管理员统一后台。将原「组织架构」「用户管理」两个独立页面合并为一个 4-tab 页面，新增角色管理、菜单管理、日志管理。仅 admin / super 角色可见（侧边栏动态渲染，详见 §3.6）。

**4-tab 结构**：

| Tab | 布局 | 功能 |
|-----|------|------|
| 用户管理 | 左树右表 | 左侧组织架构树（点击节点筛选右侧用户，树本身支持增删改节点/批量导入/成员管理）+ 右侧用户列表表格（搜索+筛选+新增+角色分配+状态管理+审批）。待审批用户高亮显示 |
| 角色管理 | 左列表右权限树 | 左侧角色列表 + 右侧菜单权限树（带 checkbox 勾选）。选中角色→右侧显示其菜单勾选状态。super 角色菜单树只读（防误操作锁死超管）。支持角色新增/编辑/删除 |
| 菜单管理 | 树形 | 菜单树展示 + 新增/编辑/删除/排序。系统菜单标记且不可删除。图标选择器（Lucide icon 列表）。菜单数据驱动侧边栏动态渲染 |
| 日志管理 | 筛选+表格+展开 | 顶部筛选栏（操作人/操作类型/目标类型/时间范围）+ 下方表格（时间/操作人/操作/目标/摘要）。点击行展开 detail（JSONB 变更快照 before→after 格式化显示）。默认最近 7 天 |

**菜单加载机制**：用户登录成功后调 `GET /api/menus` 获取当前角色可见菜单树，存入前端状态，侧边栏据此动态渲染。切换角色或修改菜单权限后需重新登录或手动刷新才生效（菜单权限变更频率极低，无需实时同步）。

**角色与菜单权限边界**：

- 角色（Role）**只控制菜单可见性**，不控制 API 级访问。后端 `require_admin` / `require_super` 守卫继续硬编码不变。
- 新增自定义角色 = 配置「该角色能看到哪些菜单」，不影响后端 API 权限判断。
- `User.role` 字段保留为字符串（`user` / `admin` / `super` 或自定义角色 code），与 `Role.code` 逻辑关联（不加 FK），零迁移成本。

### 3.5 跨项目隔离与公开

**默认规则**：项目内全透明，项目外全隔离。被邀请入项目即视为获得该项目全部数据访问权。

**手动公开**（对象级操作，在各数据页面中执行）：

| 公开范围 | 操作方式 | 结果 |
|----------|---------|------|
| 完全公开 | 对象详情 → 标记"完全公开" | 团队空间公开资产区可见 |
| 指定用户 | 对象详情 → "共享给" → 选用户 | 对方可见该对象 |

公开的对象类型：角色卡、业务地图片段、方案、知识片段。

### 3.6 角色与菜单权限模型（🆕 V2.4 新增）

> **V2.4 新增（Phase 6）**：引入角色管理 + 菜单管理，实现菜单级动态权限。决策记录见 DECISIONS.md #58 / #59 / #63 / #67 / #68。

**设计原则（方案 A — 角色 + 菜单权限，非全量 RBAC）**：

- **角色只管菜单可见性**：Role 表定义角色，RoleMenu 多对多关联菜单。新增角色 = 配置能看到哪些菜单。
- **API 权限守卫不变**：后端 `require_admin` / `require_super` 继续硬编码检查 `User.role` 字符串，不引入 API 级 RBAC。当前 1-3 人规模，菜单级权限足够覆盖需求。
- **User.role 字符串保留**：`User.role` 继续存 `user` / `admin` / `super`，`Role.code` 与之逻辑关联（不加 FK）。新建自定义角色时 `User.role` 存自定义角色 code 字符串。零迁移成本，现有用户数据不需要变更。

**三角色初始菜单关联**（种子数据，与现有硬编码行为一致）：

| 角色 code | 可见菜单 |
|-----------|---------|
| `super` | 全部菜单（关联不可修改，防锁死超管） |
| `admin` | 除登录白名单外全部菜单 |
| `user` | 作战台 4（对话/业务地图/营销地图/拜访记录）+ 文件 2（个人空间/团队空间）+ 智能体管理 |

**菜单数据模型**：`menus` 表单表自引用树（`parent_id → menus.id`），与 Organization 表同模式。内置菜单标记 `is_system=True` 不可删除。分组节点（作战台/文件/管理/设置）+ 叶子菜单（12 条）从现有 `SidebarVariantA getNavItems()` 硬编码迁移为种子行。

**菜单加载时机**：登录后一次性加载（`GET /api/menus`），存前端状态，侧边栏数据驱动渲染。详见 §2.8。

**super 角色保护**：super 角色的菜单关联始终全部可见且不可修改，防止误操作导致超管锁死。

### 7.1 候选区（Pending Area）统一规则

1. 所有 WF AI 产出先进候选区（pending）
2. 用户在对话中确认：采纳 / 驳回 / Chat 调整后再确认
3. 采纳 → 结构化数据迁移到正式区 → 对应数据页面可见 + 文件归档到个人空间
4. 驳回 → 进偏差池/驳回池，不删除
5. Chat 调整 → 循环直到采纳或驳回
6. **V2.2 新增**：草稿区 7 天自动过期，过期前提醒用户处理
7. **V2.2 新增**：整个业务地图为一个草稿单元（非逐节点），对话中增量更新草稿
8. **V2.3 新增（2026-07 回归）· 待采纳徽标找回兜底**：除对话内联卡片外，Topbar 按当前选中项目显示"N 待采纳"红点徽标，点击展开轻量列表（业务地图 / 角色卡 / 拜访记录三类 pending 草稿），支持**直接采纳**或**跳回原对话**采纳。解决用户离开"生成草稿的那条对话"之后找不到草稿、无法采纳的问题（同一对话刷新/重开时内联卡片会从历史消息重建，不会丢失；徽标补的是"哪条对话有待采纳"的找回线索）。
   - 跳回原对话依赖草稿的 `source_session_id`：业务地图草稿已具备；**角色卡 / 拜访记录草稿须补存 `source_session_id`** 才支持跳转，否则仅可直接采纳。
9. **V2.3 新增（2026-07 回归）· 采纳卡片须展示完整字段（可审阅）**：三类草稿的「待采纳」卡片统一采用"关键字段默认展开 + 其余折叠可展开"模式，确保用户**采纳前**能看到与对应数据页面详情同等的完整字段，而非仅 2-3 个摘要字段：
   - **角色卡**：默认显示头部（姓名 / 岗位 / 部门 / 角色类型中文名 / 决策权）+ 三维评分（参与度 / 影响力 / 支持度）+ 综合评分 / 等级 / 立场 + 核心顾虑 / 影响杠杆；可展开客观层 7 字段（教育 / 履历 / 性格 / 沟通偏好 / 关系 / 与我方合作 / 与竞品合作）+ 主观层（显性 KPI / 隐性诉求 / 对我方态度 / 对竞品态度 / 顾虑 / 杠杆 / **置信度 confidence**）+ 行为列表（观察 / 解读 / 建议）。
   - **业务地图**：默认按 L1-L4 分组（含各级节点数）+ 节点名列表；可展开单节点的 payload 关键字段（L3 痛点 / AI 机会、L1 五维健康等）。
   - **拜访记录**：默认显示日期 / 类型 / 参与人 / 摘要 / key takeaways；可展开完整字段。

### 7.2 Chat 调整机制

所有 WF 产出在"采纳"之前，支持通过自然语言继续对话调整：

```
WF 产出 → 对话展示候选结果（Markdown + 结构化数据预览）
      ↓
用户不满意 → 自然语言描述修改需求
      ↓
AI 基于原文 + 用户指令重新生成 → 展示 diff 对比
      ↓
用户确认 → 更新候选 → 再次问"是否采纳？"
      ↓
可多次循环 → 直到采纳或驳回
```

适用于所有 6 个独立 Skill。

### 7.3 数据页面的数据一致性

每个数据页面展示的是**该项目下所有已确认数据的并集**，单一真源：

| 页面 | 数据查询条件 |
|------|------------|
| 业务地图 | `WHERE projectId = X AND reviewStatus = 'reviewed'` |
| 营销地图 | `WHERE projectId = X AND reviewStatus = 'reviewed'` |
| 拜访记录 | `WHERE projectId = X AND reviewStatus = 'reviewed'`（VisitRecord 需审核）+ `WHERE projectId = X AND reviewStatus = 'reviewed'`（EvidenceSource） |

> **V2.2 变更**：VisitRecord 从"无需审核"改为"需审核"，与 EvidenceSource 统一走 reviewStatus 流程。

> 不同 Chat Session 的产出通过 `projectId` 关联。用户在对话中采纳 → reviewStatus 变为 reviewed → 对应页面立即可见。不存在多 Session 数据冲突问题——每个对象有唯一 ID，多次生成同类型节点会创建新版本（保留旧版本快照），不是覆盖。

### 7.4 WF07 版本管理与审核

- 每次提交保留快照版本（项目级版本快照）
- 副手提交 → Owner 审核通过 → 发布到业务地图页面（标记 published）
- Owner 提交 → 采纳后直接发布
- Owner 审核时可对比多个版本
- **V2.2 新增**：支持版本回滚——选择历史版本替换当前数据
- **V2.2 新增**：仅业务地图支持版本回溯，营销地图和拜访记录不需要

### 7.5 证据验证联动（🆕 V2.2 新增）

> **V2.2 新增**：证据与业务地图假设节点的验证联动机制。

1. 证据关联假设节点时，AI 根据证据内容和强度自动给出**建议验证状态**（如：3条强证据支持 → 建议标记为"成立"）
2. 人工确认建议后，更新节点的 `verificationStatus`
3. 假设节点被验证后（假设 vs 现状出现偏差），**偏差池自动新增**一条偏差记录
4. 现状地图数据从假设地图**自动复制**并标记验证状态，也支持增删改查

### 7.6 态度变化自动记录（🆕 V2.2 新增）

> **V2.2 新增**：新证据关联角色卡时，自动生成 stanceChangeLog 条目。

当一条证据的 `sourceRoleId` 关联到某角色卡，且证据内容暗示态度变化时：
- 系统自动在角色卡的 `stanceChangeLog` 中新增一条记录
- 记录含：日期、此前立场→当前立场、变化原因（来自证据内容）
- 顾问可在角色卡详情中手动编辑/删除自动生成的记录

### 7.7 五维健康自动计算（🆕 V2.2 新增）

> **V2.2 新增**：五维健康评分的计算触发机制。

- **计算规则**：基于总纲文档《企业数字健康分层地图 · 整体说明（V2.0）》的五维健康观测体系
- **触发时机**：对话中更新业务地图数据后，自动触发五维健康重新评估
- **手动触发**：用户可在业务地图页面手动触发"重新评估"
- **手动覆盖**：AI 算出评分后，顾问可在页面上手动修改（CRUD），手动修改的评分标记覆盖来源

### 7.8 搜索公开信息

- WF07 每次执行实时搜索（确保信息最新）
- 搜索结果存入 Material（materialType=public_info），同时归档到个人空间对应项目/资料/公开信息/
- 同一项目内已搜关键词不重复搜（增量搜索）

### 7.9 防线 1 + 防线 2

由 `consultant-defense` Plugin 实施：

- **防线 1**：道层 System Prompt 注入 — 在对话 Session 初始化时，通过 Hook 自动注入 `dao_layer.md` 全文，不可被 RAG 覆盖
- **防线 2**：PostOutputFilter — LLM 产出后做 100% 确定性指纹匹配，过滤 `never_visible` 内容

### 7.10 方法论三层使用模式

| 模式 | 适用层 | 说明 |
|------|--------|------|
| 注入（Inject） | 道层 | 永远在 System Prompt 里，不可 RAG |
| 结构化读取 | 法层 | 全量读取，不走 RAG（缺片段就错乱） |
| RAG 检索 | 器层 | 参考资料性质，Top-K 召回 |
