## 1. 文档概述

仓库中已存在 [Agent平台技术说明书V2.0.md](Agent平台技术说明书V2.0.md),那份文档的定位是**面向"AI 顾问作战台"应用层的扩展设计**(客户/项目/业务地图/营销地图/拜访记录 + 7 个 Skill + 2 个 Plugin),描述的是"**要做成什么样**"。

## 2. 平台定位与核心能力

### 2.1 平台定位

国科智能体平台是一个**基于 `claude-agent-sdk` 的多智能体对话与文件管理平台**,它把 Claude Code 的能力(对话、工具调用、Skill、Plugin、文件读写)封装成一个面向企业内部用户的 Web 产品,并提供:

- 企业微信扫码 / SSO 登录、基于 session 的全站鉴权。
- 智能体(Agent)的配置、隔离运行、技能/插件绑定。
- SSE 流式对话,支持断线恢复。
- 用户个人工作区 + 团队空间的文件管理。
- Office/PDF → Markdown 的文档转换队列。
- 使用量统计、问题反馈、登录白名单等运营能力。

### 2.2 核心能力清单

| 能力域 | 说明 | 主要代码位置 |
|--------|------|------------|
| 认证 | 企微扫码/SSO + session + 登录白名单 | `backend/app/modules/auth/`、`backend/app/api/routes/auth.py` |
| 智能体管理 | Agent CRUD、技能/插件绑定、独立工作目录 | `backend/app/modules/agents/` |
| 流式对话 | SSE 编排、运行态缓存、断线恢复 | `backend/app/modules/sessions/` |

## 4. 总体架构

### 4.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│              前端 (React 18 + TypeScript + Vite)              │
│   pages/ (对话/工作区/智能体/技能/团队空间/统计/反馈 ...)       │
│   api/client.ts  ← REST + SSE 流解析                         │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP / SSE (Cookie Session)
┌──────────────────────────┴──────────────────────────────────┐
│   API 层 (FastAPI)  api/routes/* + api/auth_guard.py         │
│   薄适配层:只做参数解析、鉴权、转发,不写业务逻辑               │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────┴──────────────────────────────────┐
│   业务逻辑层 (modules/*)                                      │
│   sessions / agents / workspace / team_spaces / conversions │
│   uploads / catalog / auth / usage / feedback                │
└──────────────┬─────────────────────────┬────────────────────┘
               │                         │
┌──────────────┴───────────┐  ┌──────────┴───────────────────┐
│  AI 运行层                │  │  存储层                       │
│  integrations/claude/     │  │  PostgreSQL (主数据)          │
│   runner / serializers    │  │  Redis (团队空间文件锁)        │
│   guard / tool_approval   │  │  本地文件系统                  │
│  integrations/mineru.py   │  │   (workspaces/agent_workdirs) │
│  integrations/openai.py   │  │                              │
└───────────────────────────┘  └──────────────────────────────┘
```

**依赖方向**:`api → modules → models/db`;`integrations` 独立,被 `modules` 调用。这套约束让路由层保持"薄",业务逻辑可独立测试。

### 4.2 部署拓扑(Docker Compose)

```
            ┌──────────────────────────────────────────┐
            │              docker network: gokagent      │
            │                                          │
            │  ┌──────────┐   ┌──────────┐             │
            │  │ postgres │   │  redis   │             │
            │  │  :5432   │   │  :6379   │             │
            │  │ health ✓ │   │ health ✓ │             │
            │  └────┬─────┘   └────┬─────┘             │
            │       │              │                     │
            │       └──────┬───────┘                     │
            │         depends_on:healthy                 │
            │          ┌───┴──────┐                      │
            │          │ backend  │ :8000  → /api/health │
            │          │ uvicorn  │  health ✓            │
            │          └──────────┘                      │
            └──────────────────────────────────────────┘
        浏览器 ──HTTP──► :8000 (后端同时托管前端 dist 静态资源)
```

三个容器全部带 healthcheck,backend 在 postgres/redis 健康后才启动(`depends_on.condition: service_healthy`)。详见第 11 章。

### 4.3 一次对话的端到端数据流

这是平台最核心的链路,也是健壮性投入最集中的地方:

```
1. 前端 EventSource 连接 POST /api/sessions/{id}/chat (流式)
2.  streaming.stream_session_chat:
     ├─ 校验用户对该 session 的访问权
     ├─ 同会话加锁(_session_lock_for)保证串行
     ├─ 创建 run(run_id),初始化 run_state(内存)
     └─ 启动后台 runner_task → runner.stream_chat
3.  runner.stream_chat:
     ├─ 解析 agent 工作目录、模型供应商
     ├─ 构造 ClaudeAgentOptions(env 注入 CLAUDE_CONFIG_DIR、
     │   permissions 白/黑名单、sandbox、allowed_tools)
     ├─ async with ClaudeSDKClient → query → receive_response
     └─ 每条 SDK 消息经 serializers 转成前端事件
4.  事件双写:
     ├─ run_state_store.append_event(seq 单调递增)  ← 断线恢复依据
     └─ 通过 asyncio.Queue 推给 SSE → 前端
5.  客户端任何时候断开:
     ├─ SSE 端置 sse_active=False,runner 不取消、继续跑
     └─ 用户重连 GET /api/sessions/{id}/running/stream?after_seq=N 续传
6.  runner 结束:
     ├─ ChatRunSummary 提取 usage/cost/stop_reason
     ├─ finalize_usage 写 usage_events(失败只记日志,不影响主链路)
     ├─ 会话消息落库 chat_sessions
     └─ finally:释放该会话持有的团队空间文件锁
```

第 8 章会逐项拆解这条链路上的健壮性设计。

### 5.3 应用生命周期(lifespan)

`backend/app/main.py:14-37` 集中了所有"启动即对齐"的动作,这是平台健壮性的第一道关口:

```python
async def _lifespan(_: FastAPI):
    app_logging.configure_logging()          # 1. 日志(幂等)
    config.apply_environment()               # 2. 配置校验 + 注入 env(fail-fast)
    await init_db()                          # 3. 幂等建表/加列/状态修复
    await ensure_all_agent_workdirs()        # 4. 补全所有 Agent 工作目录
    async with async_session() as db:
        await sync_departments(db)           # 5. 同步企微部门
    await start_conversion_task_dispatcher() # 6. 启动转换 worker 池
    await enqueue_pending_conversion_tasks() # 7. 恢复 DB 中 queued 的转换任务
    try:
        yield
    finally:
        await stop_conversion_task_dispatcher()  # 优雅关闭 worker
```

设计含义:**进程只要起来,数据库 schema、运行态、待办任务就都已对齐**,不需要额外迁移脚本或人工干预。

### 5.4 核心数据流:对话编排

见 4.3。关键代码:`backend/app/modules/sessions/streaming.py`(SSE 编排 + run 生命周期)、`backend/app/modules/sessions/run_state.py`(运行态缓存)、`backend/app/integrations/claude/runner.py`(SDK 调用)。

## 6. AI 运行时(claude-agent-sdk 集成)

### 6.1 SDK 封装结构

`backend/app/integrations/claude/` 把 claude-agent-sdk 封装成四个职责清晰的文件:

| 文件 | 职责 |
|------|------|
| `runner.py` | 构造 `ClaudeAgentOptions`、启动 SDK client、收消息、产出 `ChatRunSummary`(usage/cost/stop_reason) |
| `serializers.py` | 把 SDK 的各类消息(Assistant/User/Result/Session/System)序列化成前端 SSE 事件 |
| `guard.py` | `PreToolUse` Hook:运行时拦截 Bash 黑名单、写操作路径校验、团队空间文件锁联动 |
| `tool_approval.py` | `can_use_tool` 自动放行(交互式审批不可用,真正裁决交给 guard hook) |

调用主入口 `runner.stream_chat(...)`:`async with ClaudeSDKClient(options) as client` 保证子进程无论成功/异常都被正确回收;通过 `client.receive_response()` 异步消费消息流,支持 `stop_event` 触发 `client.interrupt()` 优雅中断。

### 6.2 智能体隔离模型

这是平台支撑**多智能体并发**的关键设计,详见 [8.2](#82-智能体并发隔离)。

### 6.3 多供应商模型路由

`backend/app/core/config.py:103-161` 实现了一套**零代码扩展**的多供应商机制:

- 新增一个兼容 Anthropic 网关协议的供应商,只需加三个环境变量:
  ```
  ANTHROPIC_PROVIDER_<NAME>_AUTH_TOKEN=...
  ANTHROPIC_PROVIDER_<NAME>_BASE_URL=...
  ANTHROPIC_PROVIDER_<NAME>_MODELS=["model-a","model-b"]
  ```
- `get_model_providers()` 扫描所有 `ANTHROPIC_PROVIDER_*_MODELS` 前缀,解析出供应商列表。
- `resolve_model_provider(model)` 按用户选的模型名定位供应商;未选则用第一个可用供应商。
- `get_available_models()` 返回扁平模型名列表给前端,**不暴露供应商 token**。
- 同时兼容旧的单供应商配置(`ANTHROPIC_AUTH_TOKEN` / `ANTHROPIC_BASE_URL` / `ANTHROPIC_MODEL`)。

### 6.4 内置 MCP(智谱 Web Search)

`runner.py` 的 `_builtin_mcp_servers()` 把智谱 Web Search MCP 作为**平台内置 MCP Server** 注入 SDK:

- 配了 `ZHIPU_WEB_SEARCH_API_KEY` → 注入 MCP,Agent 可用 `mcp__zhipu-web-search-sse__web_search` 工具。
- 没配 → 返回 `{}` **静默禁用**,不影响其它功能。

这种"未配置即降级"的模式,让搜索能力变成可选增强而非硬依赖。

## 7. 数据存储设计

### 7.1 PostgreSQL(主数据)

承载所有结构化业务数据:用户、智能体、会话、分类、技能/插件绑定、转换任务、上传任务、使用量、反馈、团队空间、登录白名单、部门缓存等。

- 引擎:`create_async_engine(database_url, future=True)`,`async_sessionmaker(expire_on_commit=False)`(`backend/app/db/session.py`)。`expire_on_commit=False` 避免 commit 后属性访问触发隐式 IO。
- 迁移策略:**用幂等 SQL 替代 Alembic**(见 [8.9](#89-数据库层健壮性)),`init_db()` 在 lifespan 启动时执行,重启即自动对齐 schema。

### 7.2 Redis(仅团队空间文件锁)

Redis 在当前平台**只有一个用途:团队空间的文件级并发锁**。不用于 session、不用于任务队列、不用于业务缓存。

- 客户端单例:`backend/app/core/redis.py`,`get_redis_client()` 全局唯一。
- 锁实现:`backend/app/modules/team_spaces/file_locks.py`,用 Lua 脚本保证获取/续期/释放的原子性(见 [8.4](#84-redis-分布式文件锁))。

> ⚠️ **生产环境没有 Redis 降级**。`fakeredis` 仅在测试中使用。Redis 不可用 = 团队空间文件锁相关写入直接报错。详见 [9.1](#91-redis-无降级)。

### 7.3 文件系统布局

```
{backend 工作目录}/
├── user_workspaces/      # 用户个人空间 (WORKSPACES_DIR)
│   └── {username}/       #   每用户一目录,Agent cwd 限定在此
│       └── .claude/skills/  # 用户自维护的内置技能
├── agent_workspaces/     # 智能体隔离工作目录 (AGENT_WORKDIRS_DIR)
│   └── {agent_code}/     #   每个 Agent 一份独立 CLAUDE_CONFIG_DIR
│       ├── CLAUDE.md
│       ├── plugins/
│       └── skills/
├── claude_data/          # 主配置目录 (CLAUDE_DATA_DIR) — 模板源
│   ├── skills/
│   ├── plugins/
```

## 8. 健壮性设计(核心)

> 本章是文档的重点。我们按"机制 → 代码位置 → 设计亮点"的方式,逐项说明底座里已经写进去的可靠性 / 安全性 / 容错设计。第 9 章会诚实地补充尚未覆盖的缺口。

### 8.1 健壮性设计原则

平台在健壮性上遵循几条一致的原则,后续每一节都是这些原则的具体落地:

1. **启动即对齐**:进程起来时,自动修复 schema、运行态、待办任务(lifespan + 幂等迁移)。
2. **状态持久化 + 内存加速**:权威状态在 DB,内存(run_state)只为流式加速,丢了能从 DB 重建会话历史。
3. **失败不静默、也不中断主链路**:异常捕获 → 记日志 → 写错误事件,但不让单点失败拖垮整条对话。
4. **纵深防御**:安全相关的检查(SDK sandbox / 工具黑名单 / 路径校验 / prompt 规则)多层叠加,不依赖单点。
5. **幂等优先**:迁移、任务领取、文件落盘都设计成可重复执行。

### 8.2 智能体并发隔离

**问题**:多个 Agent / 多个会话并发跑 claude-agent-sdk,SDK 部分行为依赖全局环境变量 `CLAUDE_CONFIG_DIR`,如果直接改 `os.environ` 会互相污染。

**机制**(`backend/app/modules/agents/workdir.py` + `backend/app/integrations/claude/runner.py`):

- 每个 Agent 在 `agent_workspaces/<agent_code>/` 下有一份**独立目录**,从主目录 `claude_data/` 拷贝 `CLAUDE.md` + 勾选的 `plugins/` + `skills/`。
- **对话主路径不走全局 env**:在 `runner.py` 构造 `ClaudeAgentOptions(env={"CLAUDE_CONFIG_DIR": str(agent_workdir), ...})`,通过 SDK 的 `env` 参数把变量注入到**子进程环境**,而非改全局 `os.environ`。

```
会话 A ──► options.env = {CLAUDE_CONFIG_DIR: /agent_workspaces/agentA}  ──► 子进程 A
会话 B ──► options.env = {CLAUDE_CONFIG_DIR: /agent_workspaces/agentB}  ──► 子进程 B
                  ↑ 互不影响,因为注入的是各自子进程环境,不是全局 os.environ
```

- 少数**必须走 SDK 全局 env** 的工具函数(读历史 / 删会话)用 `override_claude_config_dir(workdir)` 上下文管理器:`backend/app/modules/agents/workdir.py:29-46`,用一把 `asyncio.Lock` 串行化 `os.environ` 的临时修改,`finally` 恢复原值。
- 启动时 `ensure_all_agent_workdirs()` 扫描 DB 中所有 Agent,缺工作目录的补建。

**设计亮点**:把"隔离"做到了子进程级别,而不是靠加锁串行化整个进程,真正支持多会话并发。

### 8.3 流式对话的断线恢复

**问题**:LLM 一轮对话可能跑几十秒到几分钟。期间客户端网络抖动、切后台、刷新页面都会断开 SSE。如果断开就丢,用户体验极差;如果断开就取消后端 runner,又浪费已消耗的 token。

**机制**(`backend/app/modules/sessions/run_state.py` + `streaming.py`):

平台用「**运行态内存缓存 + 事件 seq + 条件变量**」方案,把 **runner 的执行**和 **SSE 的订阅**彻底解耦:

1. **`InMemoryRunStateStore`**(`run_state.py`)为每个会话当前轮缓存一个 `_RunState`:
   - 每条事件打**单调递增的 `seq`**,客户端用 `after_seq` 做断点续传。
   - **双约束防爆**:每轮最多 2000 条事件 或 300,000 字符,超限直接把状态转为 `FAILED` 并只保留一条错误事件(`_replace_with_limit_error`),防内存撑爆。
   - **run_id 防串扰**:每个新 run 生成 uuid,写入时校验 `state.run_id == run_id`,旧 run 晚到的事件不会覆盖新 run。
   - **跨事件循环通知**:用 `loop.call_soon_threadsafe` 把 runner(同步回调)的事件通知投递到 SSE 订阅者所在的 async 循环。

2. **断线恢复入口**(`backend/app/api/routes/sessions.py`):
   - `GET /api/sessions/{id}/running`:一次性返回当前运行态快照。
   - `GET /api/sessions/{id}/running/stream?after_seq=N`:SSE 恢复流。

3. **恢复流 `stream_running_session`**(`streaming.py:140-178`):
   - 先回放缓存中 `seq > after_seq` 的事件,再 `await wait_for_change(timeout=15)` 长轮询等新事件;超时即循环再取快照,天然形成心跳。
   - **run_id 切换检测**:若客户端带着上一轮的 seq 重连、而服务端已进入新一轮 run,会把 `current_seq` 归零重放当前 run 的全部事件。

4. **runner 与 SSE 解耦**(`streaming.py:194-402`):
   - runner 作为独立 `asyncio.Task` 跑,通过 `asyncio.Queue` 推事件给 SSE。
   - **SSE 断开只置 `sse_active=False`,不取消也不等待 runner**——runner 继续跑、继续写 run_state,结束时照常统计 usage、落库、释放锁。
   - 用户随时重连 `/running/stream?after_seq=N` 把漏掉的事件补上。

```
         ┌─────────────────────────────┐
runner ──┤  run_state_store (内存,seq)  ├── 持续写入,不关心有没有人订阅
         └──────────┬──────────────────┘
                    │ snapshot(after_seq)
   ┌────────────────┴────────────────┐
   ▼                                 ▼
SSE 连接 #1                    SSE 连接 #2(断线重连)
(实时推)                       (从 after_seq 回放 + 续传)
```

**设计亮点**:这是底座最值得强调的设计——**断线不丢消息、断线不浪费 token、断线不影响 usage 统计**。

> ⚠️ 局限:run_state 是**进程内内存**,不跨进程持久化。多副本部署时,断线重连需命中同一副本(粘性会话),否则恢复失败(但会话历史仍在 DB,不影响下次)。详见 [9.2](#92-run_state-为进程内内存)。

### 8.4 Redis 分布式文件锁

**问题**:团队空间里多个用户 / 多个 Agent 会话可能同时编辑同一文件,需要文件级互斥锁;且要支持锁续期(一个长会话持续写)、会话结束批量释放、过期锁抢占。

**机制**(`backend/app/modules/team_spaces/file_locks.py`):

用**三段 Lua 脚本**在 Redis 侧保证多步操作的原子性:

| 脚本 | 行为 |
|------|------|
| `LOCK_ACQUIRE_SCRIPT` | 锁不存在→`ACQUIRED`;同 `lock_token` 再入→`REENTRANT` 并刷新过期时间(**续期**);锁已过期→`TAKEN_OVER_EXPIRED`(**抢占**);否则返回 `LOCKED` + 当前持有者信息(用于冲突提示) |
| `LOCK_VALIDATE_SCRIPT` | 校验 `lock_token` 匹配且未过期 |
| `LOCK_RELEASE_SCRIPT` | 仅当 `lock_token` 匹配才 `DEL`(防误释别人的锁) |

配套设计:

- **TTL 容忍窗口**:Redis key 的 TTL = 业务过期时间 + `cleanup_grace`(默认 300s),给 GC 留余量。
- **续期靠重入**:Agent 写入工具每次触发 `_team_file_lock_hook`(`guard.py`)都调 `try_lock_file`,命中同 `lock_token` 走 `REENTRANT` 刷新过期时间,等价于心跳续期——长会话持续写同一文件不会因 TTL 过期丢锁。
- **owner 索引批量释放**:`SADD owner_index_key key` 记录每个 token 持有的所有锁;会话结束时 `release_owner_locks` 一次清掉该会话持有的全部锁(在 `streaming.py` runner 的 `finally` 里调用)。
- **Agent 会话 token**:`agent_lock_token(session_id) = "agent:" + session_id`,一个会话一个 token。
- **key 命名**:路径用 sha256 哈希(`team_space:file_lock:{space_id}:{sha256(path)}`),避免特殊字符。

**设计亮点**:用 Lua 脚本把"判断 + 写入 + 续期"做成原子操作,避免了"先 GET 再 SET"的竞态;可重入 + 自动续期 + 批量释放,贴合"一个会话连续多次写多个文件"的真实使用模式。

### 8.5 异步任务队列的可靠性

平台有两套 asyncio 任务队列(转换、上传),都遵循「**DB 持久化 + 启动重放 + CAS 领取 + 重启状态修复**」的模式。

#### 8.5.1 文档转换队列(`backend/app/modules/conversions/service.py`)

- **Worker 池**:启动时按 `MINERU_MAX_CONCURRENT_REQUESTS`(默认 2)起 N 个 `conversion-worker-{i}` 协程,共享一个 `asyncio.Queue`。
- **启动重放**:`enqueue_pending_conversion_tasks()` 把 DB 中所有 `status='queued'` 的任务按创建时间重放入队(在 lifespan 中调用)。
- **重启状态修复**(`migrations.py:240-244`):每次启动把 `status='running'` 的转换任务重置回 `queued`,清除重启残留的僵尸态。
- **CAS 领取**(`service.py:144-160`):worker 执行前用 `UPDATE ... WHERE id=? AND status IN ('queued','failed') RETURNING id` **原子领取**,保证同一任务不被多 worker 并发执行;`failed` 分支同时支持重试。
- **进程内去重**:`_scheduled_task_ids` 集合防止同一 task_id 重复入队。
- **异常处理**:`except asyncio.CancelledError: raise`(保留取消语义),其它异常记日志、写 `error_message` 回 DB,**不自动重试**(用户可手动重发)。

#### 8.5.2 文件上传任务(`backend/app/modules/uploads/tasks.py`)

上传是断点式、分片的,可靠性要求更高:

- **DB 层单 running 约束**:部分唯一索引 `uq_upload_tasks_one_running_per_user`(`ON username WHERE status='running'`)从数据库约束层保证**同一用户同时只有一个 running 上传**。
- **CAS 领取**:`UPDATE ... WHERE status='queued' RETURNING` 领取。
- **原子落盘**:`save_upload_task_file` 先写临时文件 `.uploading`,再用 `os.link` 硬链接到目标,`FileExistsError` 时标记失败——避免 TOCTOU 竞态。
- **临时文件独占**:`O_CREAT | O_EXCL` 独占创建,并发请求自动用 `.1 .2` 后缀错开。
- **断线清理**:`abandon_upload_tasks` 在页面刷新 / 客户端异常时把未完成任务标 `failed`。
- **重启兜底**(`migrations.py:299-304`):启动把 `running` 上传改 `failed` 并写"服务重启导致上传中断,请重新上传"。

#### 8.5.3 与团队空间文件锁联动

`conversions/service.py:175-197`:团队空间的转换任务执行前先 `try_lock_file`,拿不到锁直接失败并提示"文件正在被其他用户或会话编辑"——避免转换覆盖正在被编辑的文件。

**设计亮点**:不依赖外部队列服务(Celery/RQ),用"DB 是权威 + asyncio Queue 是调度 + CAS 防并发 + 启动重放 + 状态修复"的组合,在单进程内做到了**重启不丢任务、不重复执行、不残留僵尸态**。

### 8.6 路径安全与恶意输入防护

用户输入的文件路径、上传的压缩包、第三方服务(MinerU)返回的 zip,都是攻击面。平台在这一层有多重防护。

#### 8.6.1 工作区路径越权防护

`backend/app/modules/workspace/paths.py:35-56` 的 `resolve_inside_workspace(workspace, rel_path)` 是几乎所有文件操作的统一入口:

```python
def resolve_inside_workspace(workspace: Path, rel_path: str) -> Path:
    if not rel_path or rel_path.strip() == "":
        raise HTTPException(400, "路径不能为空")
    if rel_path.startswith("/") or rel_path.startswith("\\"):
        raise HTTPException(400, "只能使用工作区相对路径")
    target = (workspace / rel_path).resolve()       # 解析符号链接
    ws_resolved = workspace.resolve()
    if not target.is_relative_to(ws_resolved):      # 边界判定
        raise HTTPException(400, "路径越出工作区")
    return target
```

`Path.resolve()` 先解析符号链接,再用 `is_relative_to` 判边界——能挡住 `../` 逃逸**和**符号链接攻击。

#### 8.6.2 文件名清洗

`backend/app/core/utils.py` 的 `safe_filename` 把控制符、路径分隔符、Windows 禁用符、空白(含全角空格)替换为 `_`,还特别处理了视觉易混淆字符 `丨`(形似 `|`)和 CJK 弯引号——这是为了防止 LLM 生成工具调用时静默规整引号导致 Read/Write 失败的实战经验。

#### 8.6.3 Zip 炸弹 / 路径穿越防护

**上传插件/技能的 zip**(`backend/app/modules/catalog/zip_validator.py`):大小上限 50MB、拒绝绝对路径和 `..` 段、解压后再用 `relative_to` 兜底、目录名正则 `^[a-zA-Z0-9_\-]+$`、`BadZipFile` 捕获、SKILL.md/plugin.json 结构校验。

**MinerU 返回的 zip**(`backend/app/integrations/mineru.py:142-208`)——这是平台**最严格**的 zip 防护:

| 约束 | 值 |
|------|-----|
| 条目数上限 | 1,000 |
| 总解压尺寸 | 512 MB |
| 单文件尺寸 | 128 MB |

且基于 `ZipInfo.file_size`(压缩前已知)**解压前预判**,边写边累计、单文件超限即时中断(防声明尺寸造假),解压后再 `is_relative_to` 兜底,最后用"先 rename backup 再 rename staging"**原子发布**,避免转换中途崩溃留下半成品。

#### 8.6.4 资源耗尽防护

`workspace/paths.py` 的目录遍历有 `_MAX_DEPTH=4`、`_MAX_ENTRIES_PER_DIR=500`,并跳过 `node_modules/__pycache__/.git` 等噪声目录,防文件树接口被恶意深目录耗资源。

**设计亮点**:针对**每个外部输入源**(用户路径、用户 zip、第三方服务返回)都设了独立防护,且都遵循"预判 + 兜底"双重校验。

### 8.7 认证、会话与权限

#### 8.7.1 全局 session 守卫

`backend/app/api/auth_guard.py` 注册一个中间件,在路由挂载前拦截所有 `/api/` 请求:

- 白名单(health / 登出 / 企微登录回调 / 内部预览等约 9 个路径)免登录;OPTIONS 一律放行。
- 非白名单请求必须 `request.session["user_id"]` 存在**且 DB 能查到该 User**,否则 401 并清 session;成功则把 `user` 挂到 `request.state.user`。

这意味着:**没有一个业务 API 能绕过登录校验**。

#### 8.7.2 会话签发

- Starlette `SessionMiddleware(secret_key=settings.app_secret)`,基于签名 Cookie。
- `app_secret` 在 `config.apply_environment()` 中**强制非空**,缺失直接 `RuntimeError` 拒绝启动。
- 登录成功写 `request.session["user_id"] = user.id`;登出 `request.session.clear()`。

#### 8.7.3 角色权限

`backend/app/api/deps.py`:

- `current_user`:从 session 取 `user_id` 查 DB,用户不存在则清 session 并 401。
- `require_admin` / `require_super`:`_require_role` 判断 `user.role`,开发环境(`APP_ENV=development`)放宽以便本地调试。

#### 8.7.4 登录白名单

`backend/app/modules/auth/login_whitelist.py`:用户名 + 部门**双层白名单**。两个表都空时全放行(`empty_whitelist`);部门支持**向上继承**(祖先部门在白名单则放行,带 `seen` 集合防环)。企微登录在写本地用户前校验,未命中 403。

#### 8.7.5 OAuth state 防 CSRF

- SSO 模式:`state = secrets.token_urlsafe(32)` 存 session,回调校验 `expected_state == state`。
- 二维码模式:`state` + 服务端 `_auth_code_store`(带 5 分钟过期),`_cleanup_expired_codes` 定期清理。

#### 8.7.6 密码哈希

`backend/app/core/security.py`:passlib + bcrypt(`CryptContext(schemes=["bcrypt"])`)。企微用户用 `secrets.token_urlsafe(32)` 随机密码占位。

**设计亮点**:认证链路从"全局中间件兜底"到"依赖注入细粒度授权"到"白名单 + CSRF state + bcrypt",形成完整闭环。

### 8.8 工具调用的纵深防护(guard)

Agent 会调用 `Bash`、`Write`、`Edit`、`Read` 等工具,这些工具能力很强,必须防止被 prompt 注入后干危险的事。平台用**四层纵深防御**:

```
第 1 层  SDK sandbox / filesystem 白名单   (runner.py:193-217)  进程级隔离
第 2 层  disallowed_tools + allowed_tools   (runner.py:233-265)  声明式工具白/黑名单
第 3 层  PreToolUse Hook (guard.py)         运行时命令解析与裁决  ← 主力
第 4 层  system prompt 硬规则               (runner.py:218-230)  语义层软约束
```

**第 3 层(主力)的三个 Hook**(`backend/app/integrations/claude/guard.py`):

1. **Bash 黑名单**(`_bash_safety_hook`):
   - 黑名单含 `sudo/su/chmod/chown/dd/mkfs/curl/wget/ssh/scp/kill/reboot/shutdown/docker/kubectl/...`,以及 `sh/bash/zsh/fish`(防嵌套 shell 绕过)。
   - `_split_shell_segments` 按 `;` `&&` `||` `|` **拆段**,对每段独立检查——绕过管道拼接伪装。
   - `_command_name` 跳过环境变量赋值和 `command/builtin/exec/nohup/env` 包装,正确提取真正的命令名。
   - `_bash_command_may_write` 检测 `>`/`>>` 或写命令,只读空间下一律拒绝。

2. **Write/Edit/MultiEdit 路由**:
   - 只读空间:一律 deny。
   - 团队空间:解析 `file_path`/`notebook_path`,规范化到 workspace 内相对路径,调 `FileLockService.try_lock_file`——**拿到锁才放行**,否则 deny"文件正在被其他用户或会话编辑"。

3. **自动审批**(`tool_approval.py`):服务端无交互环境,`can_use_tool=auto_approve_tool` 只为避免 SDK 阻塞等待人工确认,**真正的安全裁决由 PreToolUse hook 负责**——注释里写得很明确。

**第 2 层(声明式)**:白名单 `Workflow/Skill/Read/Write/Edit/MultiEdit/Glob/Grep/Bash/WebFetch`;显式禁 `WebSearch`/`AskUserQuestion` + `Bash(sudo *)` 等危险模式;`permissions.deny` 含 `Read(./.env)`、`Write(/**)`、`Write({agent_workdir}/**)`(禁止写其它 Agent 目录)。

**设计亮点**:工具安全不押注在单层。声明式黑名单挡得住已知模式,运行时命令解析挡得住拼接伪装,sandbox 挡得住逃逸,prompt 规则做语义兜底。

### 8.9 数据库层健壮性

#### 8.9.1 幂等迁移(以纯 SQL 替代 Alembic)

`backend/app/db/migrations.py` 的 `init_db()` 在 `engine.begin()` 事务内一次性完成,且每一步都设计为**可重复执行**:

- 建表:`Base.metadata.create_all` + `CREATE TABLE IF NOT EXISTS`。
- 加列:每列先查 `information_schema.columns`,count=0 才 `ALTER TABLE ADD COLUMN`。
- 索引:`CREATE INDEX IF NOT EXISTS`。
- 唯一约束:先查 `information_schema.table_constraints` count 再加。
- 部分唯一索引:`CREATE UNIQUE INDEX ... WHERE status='running'`(强制同用户单 running 上传)。
- 数据回填:如 `agents.code` 新增列时 `UPDATE agents SET code='agent-'||id WHERE code=''`。

**取舍说明**:用幂等 SQL 换来了"无需 Alembic 版本管理、重启即自愈",代价是迁移逻辑较冗长、且全在一个事务里(任一 SQL 失败整体回滚 → 启动失败)。在当前规模下,这个取舍是划算的。

#### 8.9.2 启动态状态修复

迁移时顺手把"上次未完成的运行态"修好:

- 转换任务 `running → queued`(重启后可被重新调度)。
- 上传任务 `running → failed`(带友好提示,让用户重传)。
- 清理无 Agent 的旧会话 `DELETE FROM chat_sessions WHERE agent_id IS NULL`。
- 插入默认分类、默认 Agent。

#### 8.9.3 事务管理

- DDL 自动事务(`engine.begin()`)。
- 业务层显式 `await session.commit()`;任务领取用 `UPDATE ... RETURNING` 配合 commit 做 CAS。
- `IntegrityError` 捕获 rollback。

> ⚠️ 缺口:`create_async_engine` 未显式配置 `pool_pre_ping` / `pool_recycle`,长连接可能因 DB 端断连报错。详见 [9.5](#95-db-连接池未显式调优)。

### 8.10 SDK 集成的容错

`backend/app/integrations/claude/runner.py` + `streaming.py:323-374`:

- **客户端生命周期**:`async with ClaudeSDKClient(options) as client`,保证子进程无论成功/异常都被回收。
- **优雅中断**:`stop_event` 被置位时调 `client.interrupt()`(用户点"停止"),而非强杀。
- **异常全链路兜底**:`stream_chat` 自身不捕获异常,向上抛到 `streaming.runner()` 的 `except Exception`:
  - 记 `logger.exception`;
  - 写错误事件到 SSE + run_state;
  - 调 `finalize_usage(None, error_message)` **仍记 usage**(失败也统计);
  - `finally` 释放团队空间文件锁。
- **结构化 summary**:`ChatRunSummary` 从 `ResultMessage` 提取 `is_error/stop_reason/usage/cost/duration`,即使 `result_message is None`(异常路径)也返回结构化对象供统计使用。
- **缓冲上限**:`max_buffer_size=2*1024*1024` 防 SDK 内部超大消息撑爆内存。
- **runner task 异常消费**:`add_done_callback` 取出未读异常记日志,避免"Task exception was never retrieved"。
- **序列化容错**:`serializers.py` 对多种消息类型分支处理,字段缺失用 `.get(key, default)` 兜底。

**设计亮点**:SDK 的任何异常都不会让对话"卡死"或"静默失败"——要么转成用户可见的错误事件,要么至少在 usage / 日志里留下痕迹。

### 8.11 配置的 fail-fast 校验

`backend/app/core/config.py:164-198` 的 `apply_environment()` 在 lifespan 首批执行(launch SDK 前):

- 强校验 `APP_SECRET` / `ANTHROPIC_AUTH_TOKEN`(或默认供应商)/ `ANTHROPIC_BASE_URL` / 企微三件套,任一缺失直接 `RuntimeError` **拒绝启动**。
- 数值配置用 `Field(ge=1)` 等约束边界(`mineru_max_concurrent_requests`、文件锁 TTL)。
- 启动时 `mkdir -p` 所有数据目录。
- `get_settings()` 用 `@lru_cache` 单例化,避免重复读 `.env`。

**设计意图**:**配置错误在启动当场暴露**,绝不带病运行。

### 8.12 部署健壮性

`docker/docker-compose.yml`:

- **全容器 healthcheck**:postgres(`pg_isready`)、redis(`redis-cli ping`)、backend(`curl /api/health`)。
- **依赖顺序**:`backend` 在 postgres/redis `service_healthy` 后才启动,避免启动期连接失败。
- **start_period**:`backend` healthcheck 给 30s 启动宽限期。
- **持久化**:postgres data、redis AOF(`--appendonly yes`)、各工作目录全映射到 volume,容器重建不丢数据。
- **restart: always**:进程 / 容器异常自动拉起。
- **优雅关闭**:lifespan 的 `finally` 停掉转换 worker,避免任务半途中断。

### 8.13 错误处理与日志

- **结构化滚动日志**(`backend/app/core/logging.py`):`RotatingFileHandler`(10MB × 5 份,UTF-8)+ 控制台;默认 `LOG_LEVEL=ERROR` 控制生产噪音;`configure_logging(force=False)` 幂等;环境变量驱动(`LOG_LEVEL/LOG_FILE/LOG_MAX_BYTES/LOG_BACKUP_COUNT`),非法值回退默认。
- **统一异常模式**:全系统遵循"捕获 → 记日志 → 写错误事件 → 不中断主链路"。代表实现:`streaming.runner()`、`finalize_usage`(usage 持久化失败只记日志)、`_conversion_worker`、语义标题生成(fallback 本地截断)。

> ⚠️ 缺口:未注册 FastAPI 全局 `exception_handler`,未捕获异常走默认 500。详见 [9.6](#96-缺少全局异常处理器)。

## 9. 已知风险与改进方向

本章诚实列出当前底座**尚未覆盖**的健壮性缺口,供技术评审与后续迭代参考。排序大致按影响面从大到小。

### 9.1 Redis 无降级

- **现状**:生产代码中 `get_redis_client()` 总是连真实 Redis,没有 fakeredis 回退。Redis 宕机 → 团队空间文件锁相关写入(团队空间编辑、团队空间转换)**直接报错**。
- **影响**:Redis 是团队空间协作的硬依赖。
- **建议**:短期——监控 Redis 健康,docker-compose 已有 healthcheck;中期——评估是否对锁失败做有限重试或只读降级。

### 9.2 run_state 为进程内内存

- **现状**:`InMemoryRunStateStore` 是进程内缓存,不跨进程持久化。
- **影响**:多副本部署时,SSE 断线重连需命中同一副本(粘性会话),否则恢复失败。会话历史仍在 DB,不影响"下次打开看到历史"。
- **建议**:若走向多副本,把 run_state 迁移到 Redis(已有 Redis 基础设施),或在网关层做粘性路由。

### 9.3 SDK 调用无显式超时

- **现状**:`runner.stream_chat` / `receive_response` 没有 `asyncio.wait_for` 兜底。
- **影响**:若 SDK 卡死,只能靠用户手动点"停止",否则该 run 的资源(锁、内存)会一直占用。
- **建议**:加一个合理的整体超时(如 10 分钟)`asyncio.wait_for` 兜底,超时后 `interrupt()` 并标记 run 失败。注意:OpenAI 兼容调用、MinerU、语义标题生成都已显式设了 timeout,这里是个不对称的缺口。

### 9.4 Cookie 安全属性未显式加固

- **现状**:`SessionMiddleware` 默认 `SameSite=lax`、Cookie 自动 HttpOnly,但未显式开启 `Secure`。
- **影响**:非 HTTPS 链路下 cookie 可能被中间网络读取。
- **建议**:生产部署配合 HTTPS 反代,并在 `SessionMiddleware` 显式 `https_only=True`(Secure)。

### 9.5 DB 连接池未显式调优

- **现状**:`create_async_engine` 未设 `pool_size`/`max_overflow`/`pool_recycle`/`pool_pre_ping`,用 SQLAlchemy 默认(pool_size=5, max_overflow=10)。
- **影响**:长连接 + PostgreSQL 场景,若 DB 端主动断连,池中 stale 连接可能报错。
- **建议**:生产加上 `pool_pre_ping=True`、`pool_recycle=<略小于 DB wait_timeout>`,并按并发量评估 `pool_size`。

### 9.6 缺少全局异常处理器

- **现状**:`main.py` 未注册 FastAPI `exception_handler`,未捕获异常走默认 500。
- **影响**:偶发的未捕获异常只产生默认 500 响应,不一定被结构化日志记录,排障困难。
- **建议**:补一个全局 `exception_handler`,把 500 也写成结构化日志 + 统一错误响应体。

### 9.7 任务失败无自动重试

- **现状**:转换 / 上传任务失败只记 `error_message`,不自动重试(依赖用户手动重发,或下次启动被 CAS 的 `failed` 分支捡起)。
- **影响**:瞬时故障(MinerU 短暂不可用)会直接表现为用户可见失败。
- **建议**:为转换任务引入有限次(如 3 次)指数退避重试,需配合幂等性校验。

### 9.8 迁移无分段提交

- **现状**:`init_db()` 全在一个 `engine.begin()` 事务里。
- **影响**:任一条 SQL 失败 → 整体回滚 → 启动失败(原子性是优点,但失败面集中)。
- **建议**:迁移规模继续增长时,考虑按"建表 / 加列 / 数据回填 / 状态修复"分段提交,降低单点失败影响面。

- 容器内数据目录统一映射到 `./volumes/*`,便于备份与迁移。

### 11.4 健康检查与可观测性

- `/api/health` 端点 + docker healthcheck。

## 12. 扩展性设计

底座在以下几个点预留了低成本的扩展路径:

| 扩展场景 | 扩展方式 | 是否需改核心代码 |
|---------|---------|----------------|
| 新增 LLM 供应商 | 加 `ANTHROPIC_PROVIDER_<NAME>_*` 三个环境变量 | 否 |

| 新增 Agent 技能/插件 | 管理端上传 → catalog 扫描注册 → 绑定到 Agent | 否(走管理 API) |
| 新增业务模块 | 按 `api/routes → modules → models/schemas` 分层加,`router.py` 注册 | 否(遵循既有分层) |
| 新增转换后处理 | 在转换 worker 内挂 hook | 小 |
| 表结构变更 | 在 `migrations.init_db()` 加幂等 SQL(查 `information_schema` 再加列/索引) | 小 |
