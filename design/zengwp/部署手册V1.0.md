## 一、整体架构

```
                         ┌──────────────┐
        浏览器 ─── HTTPS ─▶│  Nginx(可选) │── 80/443 ─┐
                         └──────────────┘            ▼
                                            ┌──────────────────┐
                                            │   backend :8000   │  FastAPI + uvicorn
                                            │  (Python 3.13)    │  ├─ /api/* 业务接口
                                            └────────┬──────────┘  ├─ /api/health 健康检查
                                       ┌─────────────┼─────────────┘  └─ StaticFiles 托管前端 dist
                                       ▼             ▼
                                ┌──────────┐   ┌──────────┐
                                │PostgreSQL│   │  Redis   │  团队空间文件锁
                                │   16     │   │   7      │
                                └──────────┘   └──────────┘

  backend 运行时还会调用下列「外部依赖」（部署前务必就绪）：
    ① Claude Code CLI（claude-agent-sdk 拉起子进程，必装）
    ② LLM 供应商 API（GLM / DeepSeek / MiniMax / Anthropic 兼容网关，必填 key）
    ③ MinerU 文档转换服务（可选，上传 Office/PDF 时需要）
    ④ 企业微信（V2.2 后认证已弃用，但启动校验仍要求填配置）
```

**组件清单**

| 组件 | 技术 | 作用 | 必须 |
|------|------|------|------|
| backend | Python 3.13 + FastAPI + uvicorn + SQLAlchemy 2.0(async) | 业务 API + SSE 流式对话 + 托管前端静态 | ✅ |
| frontend | React 18 + Vite（构建为静态 dist） | Web 前端，构建产物由 backend 托管 | ✅ |
| PostgreSQL 16 | 关系库 | 会话/客户/项目/地图/角色卡/拜访/草稿/审计等全部业务数据 | ✅ |
| Redis 7 | 缓存/锁 | 团队空间并发文件锁（`TEAM_SPACE_FILE_LOCK_*`） | ✅ |
| Claude Code CLI | Node.js（`@anthropic-ai/claude-code`） | claude-agent-sdk 通过子进程调用，AI 对话核心 | ✅ |
| LLM 供应商 | 外部 API | GLM(智谱)/DeepSeek/MiniMax/Anthropic 兼容网关 | ✅ |
| MinerU | 外部 HTTP 服务 | Office/PDF → Markdown 转换（上传资料时） | ⚠️ 可选 |
| LibreOffice | 系统包 | Office 预览/转换辅助（base 镜像已装） | 镜像内已含 |
| kkFileView | 外部服务 | Office 在线预览（内网，可选） | ⚠️ 可选 |

> **前端无需独立部署**：`vite build` 产物 `frontend/dist` 在镜像构建时拷入 backend，由 FastAPI `StaticFiles` 直接托管（`app/main.py` 末尾 mount）。如需 HTTPS/大并发，建议前置 Nginx（见第七章）。

## 二、环境与依赖要求

### 2.1 操作系统
- **Linux x86_64**（生产推荐）：Ubuntu 22.04/24.04、Alibaba Cloud Linux 3、CentOS 7+/Anolis 8（Docker 镜像基于 Debian 12 bookworm）。
- Windows/macOS 仅建议开发联调（Windows 下 uvicorn `--reload` 会导致 Claude 子进程拉起失败，见第九章）。

### 2.2 软件依赖
| 软件 | 版本 | 说明 |
|------|------|------|
| Docker Engine | ≥ 24.0 | 含 Compose v2（`docker compose`） |
| Docker 镜像加速 | 推荐 | 内网/国内拉取 `postgres:16.13`、`redis:7.4`、`python:3.13-slim`、`node:22`（compose 已用 `docker.1ms.run` 镜像源） |
| Claude Code CLI | 最新稳定版 | **base 镜像未自动安装，需手动装**（见第六章①） |

### 2.3 外部服务 / 凭证清单（部署前准备）
| 项 | 必填 | 获取方式 |
|----|------|---------|
| LLM 供应商 AUTH_TOKEN + BASE_URL + MODELS | ✅ | 至少一组：推荐智谱 GLM（`https://openapi.cloud-ai.cn`），或 DeepSeek（`/anthropic` 兼容端点） |
| 会话标题/意图分类 Key（DeepSeek OpenAI 兼容） | 推荐 | 复用 DeepSeek token，用于语义标题生成 |
| Claude Code CLI 登录凭证 | ✅ | `claude` 首次运行需登录（OAuth 或 API key），子进程继承 |
| 企业微信 CORP_ID / AGENT_ID / SECRET | ✅* | *V2.2 后登录已弃用企微，但 `apply_environment()` 启动仍校验非空，填任意非空占位即可 |
| 智谱 Web Search Key（ZHIPU_WEB_SEARCH_API_KEY） | ⚠️可选 | 配置后给 Agent 内置联网搜索 MCP |
| MinerU 服务地址（MINERU_API_URL） | ⚠️可选 | 默认 `http://192.168.125.11:8000/file_parse`（内网，公网部署必须改成可达地址或自建） |
| kkFileView 地址（KKFILEVIEW_BASE_URL） | ⚠️可选 | Office 在线预览，内网访问 |

## 三、Docker 一键部署（客户/生产推荐）

采用 `docker-compose.release.yml`：PostgreSQL + Redis + 预编译 release 镜像（后端经 Nuitka 编译，源码受保护；前端 dist 打入镜像）。

### 3.1 准备 release 镜像（在构建机执行一次）

```bash
cd agent-platform-main

# 1) 后端基础镜像（含 Python3.13 / Node22 / LibreOffice / Playwright 依赖 / bun / uv）
docker build -f docker/Dockerfile_base       -t gokagent-backend:base   .

# 2) release 镜像（前端构建 + Nuitka 编译后端 + 打包）
docker build -f docker/Dockerfile.release    -t gokagent-backend:release .
```

> ✅ **Claude Code CLI 已在 `Dockerfile_base` 预装**（`npm install -g @anthropic-ai/claude-code`，构建期 `claude --version` 校验，装失败即构建失败）。运行时仍需完成 `claude` 登录：容器首次启动后登录一次，或把已登录的 `~/.claude` 凭证挂进容器。

交付方式二选一：
```bash
# A. 推送私有仓库
docker tag gokagent-backend:release registry.example.com/gokagent/backend:1.0.0
docker push registry.example.com/gokagent/backend:1.0.0

# B. 离线 tar 交付
docker save gokagent-backend:release -o gokagent-backend-release.tar
```

### 3.2 客户侧部署

服务器只需：`docker-compose.release.yml` + `.env` +（私有仓库权限 或 离线 tar）。

```bash
# 放置 compose 与 .env
mkdir -p ~/gokagent && cd ~/gokagent
# 拷入 docker-compose.release.yml 与 .env（按第四章填写）

# 离线镜像载入（如用 tar）
docker load -i gokagent-backend-release.tar

# 拉起
GOKAGENT_IMAGE=gokagent-backend:release \
docker compose -f docker-compose.release.yml up -d

# 查看状态（三个容器应 healthy）
docker compose -f docker-compose.release.yml ps
```

### 3.3 创建首个管理员

```bash
docker exec -it gokagent-backend uv run python -m app.scripts.create_user <用户名> <密码> --role super
```
登录后在「系统设置 → 用户」审批/管理。

```ini

DATABASE_URL=postgresql+asyncpg://postgres:<强口令>@postgres:5432/gokagent

# LLM 至少一组（多供应商按前缀追加）

# DeepSeek 同时用于会话标题/意图分类（OpenAI 兼容端点）
ANTHROPIC_PROVIDER_DEEPSEEK_AUTH_TOKEN=<deepseek token>
ANTHROPIC_PROVIDER_DEEPSEEK_BASE_URL=https://api.deepseek.com/anthropic
ANTHROPIC_PROVIDER_DEEPSEEK_MODELS=["deepseek-v4-flash","deepseek-v4-pro"]

# 企业微信（V2.2 后登录已弃用，但启动校验要求非空，填占位即可）
WECHAT_WORK_CORP_ID=placeholder
WECHAT_WORK_AGENT_ID=placeholder
WECHAT_WORK_SECRET=placeholder
WECHAT_WORK_LOGIN_MODE=sso

# ===== 可选 =====
ZHIPU_WEB_SEARCH_API_KEY=                  # 内置联网搜索 MCP
MINERU_API_URL=http://<你的MinerU地址>/file_parse
MINERU_MAX_CONCURRENT_REQUESTS=2
APP_ENV=production
```

> 容器内 `DATABASE_URL` / `REDIS_URL` 用服务名 `postgres` / `redis`（compose 已默认覆盖）。目录类（`WORKSPACES_DIR` 等）compose 已映射到 `/app/*`，一般无需改。

## 七、反向代理与 HTTPS（生产推荐）

backend 直接对外可行，但生产建议前置 Nginx：

```nginx
server {
    listen 443 ssl http2;
    server_name your.domain.com;
    ssl_certificate     /etc/nginx/cert.pem;
    ssl_certificate_key /etc/nginx/key.pem;

    client_max_body_size 500m;          # 文档上传，按需调大
    proxy_read_timeout   600s;          # SSE 长连接，必须调大

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    # SSE：禁用缓冲，开启分块
    location ~ ^/api/sessions/.+/(chat|running/stream) {
        proxy_pass http://127.0.0.1:8000;
        proxy_buffering off;
        proxy_cache off;
        chunked_transfer_encoding on;
    }
}
```
> uvicorn 已带 `--proxy-headers --forwarded-allow-ips '*'`（release CMD），Nginx 前置可直接识别真实 IP/协议。

## 九、常见问题与已知坑

| 现象 | 根因 | 解决 |
|------|------|------|
| `Failed to start Claude Code` | Claude CLI 未登录（镜像已预装）/ Windows `--reload` 导致 SelectorEventLoop 不支持子进程 | 容器内完成 `claude` 登录或挂载 `~/.claude` 凭证；Linux 生产不带 `--reload`（默认即可） |
| 启动报 `缺少必需的企业微信环境变量` | `apply_environment()` 仍校验 WECHAT_WORK_* 非空 | `.env` 填任意非空占位值 |
| 上传 Office/PDF 报 `MineruConversionError: ... connection attempts failed` | MinerU 服务不可达（默认内网地址） | 改 `MINERU_API_URL` 为可达地址，或自建 MinerU |
| 会话标题/意图分类卡顿或超时 | DeepSeek 标题生成走外网 | 确认 DeepSeek token 与网络；或接受本地兜底标题 |
| AI 回复被 Sandbox 告警 | claude-agent-sdk 在原生 Windows 不支持沙箱 | Linux 部署无此问题；Windows 仅开发 |
| 草稿采纳卡片切页后消失 | （V2.x 已修复，见 [测试报告V1.0.md](测试报告V1.0.md) 第十章） | 已通过 `/pending-drafts` 找回 |

## 十一、阿里云服务器规格评估

### 11.1 资源消耗驱动因素

本平台**最吃内存的环节是 AI 对话**：每发起一个会话，`claude-agent-sdk` 会拉起一个 **Claude Code CLI 子进程**（Node.js），单个约占 **150–300 MB**。并发对话数 = 并发子进程数，是选型的核心变量。

| 组件 | 内存（基线） | 备注 |
|------|------------|------|
| OS + Docker | ~0.5 GB | |
| PostgreSQL 16 | ~0.4 GB（本应用数据量不大） | 业务数据：会话/地图/角色卡/拜访/草稿/审计 |
| Redis 7 | ~0.1 GB | 仅文件锁 |
| backend（Python） | ~0.5–0.8 GB | uvicorn + SQLAlchemy + 沙箱库加载 |
| **每个并发 AI 对话** | **+0.25 GB**（Claude 子进程） | **主变量** |
| MinerU（若同机） | +1–2 GB（转换时峰值） | 建议独立部署或用小规格独占 |
| 文档转换瞬时 | +0.2 GB | LibreOffice 瞬时进程 |

**估算公式**：`内存 ≈ 1.8 GB(基线) + 0.25 GB × 并发对话数 + (MinerU 同机 ? 1.5GB : 0)`

### 11.2 分档推荐

> 假设：内部顾问团队工具，工作时段集中使用，AI 对话为主要负载。规格为「**ECS（应用+DB+Redis 同机）**」一体化部署。

| 档位 | 典型规模 | 并发对话 | 推荐 ECS 规格 | 内存实算 |
|------|---------|---------|--------------|---------|
| **POC / 小团队** | ≤ 15 用户 | ≤ 5 | **2 vCPU / 8 GiB**（如 `ecs.g8i.large`、`ecs.g7.large`） | 1.8 + 5×0.25 ≈ 3.1 GB，8G 余量充足 |
| **中型（推荐起步）** | 15–50 用户 | ≤ 10 | **4 vCPU / 16 GiB**（如 `ecs.g8i.xlarge`、`ecs.g7.xlarge`） | 1.8 + 10×0.25 ≈ 4.3 GB，16G 抗峰值/转换/扩容 |
| **中型偏重 / 含 MinerU** | 30–80 用户 | ≤ 15 | **8 vCPU / 32 GiB**（`ecs.g8i.2xlarge`） + **MinerU 独立小规格机** | 主机留余量给并发与转换 |
| **大型 / 正式生产** | ≥ 100 用户 | > 15 | 应用 8C16G×N（负载均衡）+ **RDS PostgreSQL 管理版** + Redis 云版 + MinerU 独立 | 拆分 DB，备份/HA |

### 11.3 明确推荐

**如果不确定规模，按「中型起步」选 `ecs.g8i.xlarge`（4 vCPU / 16 GiB）**——理由：
1. Claude 子进程是内存大户，并发一上来 8G 会偏紧，16G 留足峰值与文档转换余量；
2. 4 vCPU 够 backend + PG + Redis + 并发子进程调度；
3. 后续若用户增长，可平滑升配或拆 RDS，无需重构。

**极小预算 / 纯 POC（≤3 并发、不上传大文档）**：`ecs.g8i.large`（2C8G）甚至经济型 `ecs.u1-c1m2.large`（2C4G，**紧张**，仅建议临时验证）。

### 11.4 配套规格

| 项 | 建议 |
|----|------|
| **系统盘** | ESSD PL1 或高效云盘，**40–100 GB**（系统+镜像+应用+用户工作区）。用户上传文件多时按 100GB 起，可扩容。 |
| **数据盘** | 如规模较大，独立 ESSD 挂载给 `volumes/postgres_data` 与 `volumes/user_workspaces`，便于扩容与备份。 |
| **公网带宽** | 内部团队：**按使用流量计费，带宽上限 5–10 Mbps** 即可（SSE 为小文本流；LLM 响应是入站流量，免费）。对外正式：固定 5 Mbps 起。 |
| **地域** | 选离 LLM 供应商与用户都近的国内地域（华东1-杭州 / 华北2-北京 / 华东2-上海）。GLM、DeepSeek 均在国内，时延低。 |
| **镜像** | Ubuntu 22.04/24.04 或 Alibaba Cloud Linux 3（装 Docker 即可）。 |
| **安全组** | 仅放行 80/443（或 8000）；**PostgreSQL 5432 / Redis 6379 不要对公网开放**，仅容器内网/本机访问。 |

### 11.5 生产进阶建议

- **数据库托管**：用户 > 50 或有数据安全要求时，用 **RDS PostgreSQL 16**（高可用 + 自动备份），ECS 只跑应用，规格可降。
- **对象存储**：大量用户上传文件可后续迁移到 **OSS**（当前实现是本地卷，需改代码）。
- **HTTPS 证书**：阿里云 SSL（免费 DV）+ Nginx，或用 ALB/CLB 监听转发。
- **监控告警**：云监控盯 ECS 内存/CPU + RDS；内存接近 80% 告警（Claude 子进程易拉高）。
- **成本优化**：长期稳定负载买**包年包月**（约 3–5 折）；POC 用**按量付费**。

### 11.6 一句话结论

> **内部团队首选：华东/华北地域，`ecs.g8i.xlarge`（4 vCPU/16 GiB）+ 100 GB ESSD + 按流量 5 Mbps，Linux 系统，Docker compose release 一键部署。预算紧的 POC 可降到 `ecs.g8i.large`（2 vCPU/8 GiB）。**
