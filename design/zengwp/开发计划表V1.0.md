## 一、工期方法论

V1.0 按传统"人工全做"估算 ~59.5 人天。V1.1 改为 Claude Code 主导执行，**人工只负责决策、验收、领域校准**：

| 工作类型 | V1.0 人工占比 | V1.1 模型 | 缩减倍数 |
|---------|--------------|----------|---------|
| 机械脚手架（DDL/ORM/Schema/CRUD/类型） | 100% 人工 | Claude 批量生成，人工审 15-20% | **5-6×** |
| 业务逻辑（权限/自动 Agent/归档） | 100% 人工 | Claude 生成骨架，人工审逻辑 | **3×** |
| 前端页面（基于原型重构） | 100% 人工 | Claude 重构对接，人工审交互 | **2-3×** |
| Skill Prompt（领域质量） | 100% 人工 | Claude 起草，人工**迭代校准**（瓶颈） | **1.5-2×** |
| 集成调试 | 100% 人工 | Claude 定位修复，人工验收 | **2×** |

> ⚠️ **工期下限由 Skill Prompt 质量决定**，不是编码速度。WF07（跨层一致性 + L3 本体抽取）是最难压缩的卡。

**总工期**：约 **100 人时 ≈ 12-13 工作日**（人工主导 Claude 执行），较 V1.0 的 ~59.5 人天缩短约 **58%**。

## 二、如何用 Claude Code 执行本计划

每张 `CC-XX` 任务卡都是**可直接喂给 Claude Code 的 prompt 骨架**。推荐工作流：

1. **按依赖顺序逐卡执行**（见第十章关键路径），不要跳序。
2. **每卡执行步骤**：把卡的"Claude 输入 + 目标文件 + 验收"贴给 Claude Code → Claude 实现 → 跑"验证命令" → 人工对照"验收"勾掉。
3. **每卡完成后提交一次**（遵循 `feat:/fix:` 规范），保持可回滚。
4. **到达里程碑门禁**（第十二章）才进下一 Phase。

> 任务卡字段说明：
> - **Claude 输入**：Claude 实现前需要读的基准文档章节 / 参考代码
> - **目标文件**：Claude 将创建或修改的具体路径
> - **验收**：可勾选的完成标准
> - **验证命令**：客观判定通过的命令（`pytest` / `npm run build` / 手动跑）
