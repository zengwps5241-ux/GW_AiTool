## Problem Statement

`design/jason/tools-platform-prd.md` 要求在现有智能体平台中新增工具管理平台，统一承载 MCP Server、HTTP API、用户上传代码生成的托管工具和 INTERNAL 内部能力。核心不是做一个工具列表，而是把 Agent 调用外部能力时的权限、`user_id` 透传、降级、稳定性拦截、调用统计放到同一条执行链路里。

本版选型更新了一个关键前提：**允许引入 Kubernetes/K3s**。因此 Serverless/FaaS 候选不再受限于 Docker Compose 或单机方案，K8s-native 的弹性伸缩、scale-to-zero、网络隔离和多租户资源治理可以进入首选方案。

技术路线保留两条：

- **路线 A：自研 Tools Gateway + 集成现成 FaaS/Serverless 平台**
- **路线 B：集成现成 Gateway + 集成现成 FaaS/Serverless 平台**

现成 Gateway 重点评估 **agentgateway**。FaaS/Serverless 平台要求：开源、仍在维护、支持多语言，至少 Java/Python/JS，并支持弹性伸缩。

## Current System Context

现有系统的关键集成点：

- 后端是 FastAPI + SQLAlchemy async + Postgres + Redis，当前本地部署形态是 `docker-compose`。
- Agent 运行时由 `backend/app/integrations/claude/runner.py::stream_chat()` 封装 `claude-agent-sdk`。
- 当前 MCP 在 runner 中通过 `_builtin_mcp_servers()` 注入，`allowed_tools` 和 `mcp_servers` 在每轮对话构造。
- 会话流在 `backend/app/modules/sessions/streaming.py::stream_session_chat()`，这里会收集 `tool_use` / `tool_result` 并在收尾时写 usage。
- 权限入口是 session-based `current_user()`，用户模型已有 `role`、企微部门字段，团队空间已有 owner/member/reader/editor 与 workspace scope。
- Claude 工具权限现在通过 `backend/app/integrations/claude/guard.py::build_pre_tool_use_hooks()` 做 Bash、写文件、团队文件锁拦截。
- Usage 统计已有 `backend/app/modules/usage/service.py::persist_usage_event()`，后续可扩展工具调用画像。

这意味着工具平台不能让 Agent 直接连外部工具后端。正确的集成点是：平台控制面先按用户、Agent、团队空间编译可用工具集，再由 runner 注入统一 MCP endpoint；所有真实调用都经网关和 AuthGateway。

## Constraints

- 允许引入 Kubernetes/K3s。首版建议优先 **K3s**，降低客户现场部署复杂度，同时保留迁移标准 Kubernetes 的空间。
- 网关权限必须复用平台现有用户、角色、团队空间和会话上下文。
- Agent 不能直连外部 MCP/API/函数实例绕过平台权限。
- 用户上传代码按不可信代码处理，不能直接运行在 backend 容器里。
- K8s/K3s 不是完整安全沙箱。仍需要 namespace、ServiceAccount、NetworkPolicy、ResourceQuota、PodSecurity、seccomp/AppArmor、只读根文件系统、非 root 用户和出向白名单。
- Serverless 平台负责运行时弹性和生命周期，平台 backend 仍负责 ConnectorRegistry、权限真相、审计、包管理、业务级调用日志。
- 选型优先开源与可维护性；如果核心弹性能力依赖商业版，则不能作为首选。

## Agentgateway Suitability

### Research Snapshot

调研对象：`agentgateway/agentgateway`

- 定位：面向 AI Agent 的开源代理，覆盖 MCP、A2A、LLM、普通 HTTP/gRPC 流量。
- 语言与许可：Rust，Apache-2.0。
- 维护状态：GitHub API 返回约 3.1k stars、523 forks，2026-06-09 仍活跃；release 有 `v1.3.0-alpha.1`，稳定 tag 已到 `v1.2.1`。
- 部署：支持 standalone binary、Docker、Docker Compose，也支持 Kubernetes controller 和 Gateway API。
- MCP 能力：支持 MCP Gateway、stdio/HTTP/SSE/Streamable HTTP transport、MCP federation、Virtual MCP、OpenAPI to MCP。
- 鉴权能力：支持 JWT、API key、OAuth、Basic、External Authorization、HTTP authorization、MCP authorization。
- 策略能力：MCP authorization 支持 CEL 表达式，并能读取 `mcp.tool.name`、`mcp.tool.arguments`、`mcp.tool.target`、`mcp.methodName`、`mcp.sessionId`。
- 外部鉴权：支持 HTTP 或 Envoy External Authorization gRPC，把授权决策交给外部服务，例如平台自己的 AuthGateway。
- 可观测性：支持 OpenTelemetry metrics/logs/tracing。

### Fit For This Project

agentgateway 比 APISIX/Kong 更贴合本系统的工具网关诉求。原因是它原生理解 MCP，而本项目的核心入口正是 Claude Agent SDK 的 `mcp_servers` 和 MCP tools。传统 API Gateway 需要额外桥接 MCP tool name、tool arguments 和 tools/call 上下文；agentgateway 已经把这些作为一等对象暴露给策略层。

适合承担的职责：

- 作为路线 B 的统一工具数据面。
- 聚合外部 MCP Server、由 OpenAPI 生成的 MCP 工具、普通 HTTP API、Serverless 函数入口。
- 做 MCP tool 级 allow/deny、参数级策略、JWT/API key/OAuth、rate limit、retry、timeout、OpenTelemetry。
- 在 Kubernetes/K3s 模式下通过 Gateway API/HTTPRoute 等资源承载路由配置。
- 通过 external authorization 调用平台 AuthGateway，复用本系统用户、团队空间、Agent 会话、`user_tool_grants`。

不适合承担的职责：

- 不应替代本系统的 ConnectorRegistry。
- 不应成为唯一权限真相。平台仍要保存 ACL、用户授权、团队空间、审计日志和权限预览。
- 不负责用户代码运行。Serverless/FaaS 平台仍需单独选择。
- 不天然理解本系统当前 session cookie，需要平台给 Agent 调用链路注入短期 JWT、API key 或通过 external auth 传递会话上下文。

### Verdict

**agentgateway 合适，且应作为路线 B 的首选 Gateway。**

但推荐落地方式不是“把权限全写进 agentgateway 配置”，而是：

- agentgateway 做 MCP/HTTP 数据面与协议治理。
- 平台 FastAPI 做控制面和权限真相。
- agentgateway 的 MCP authorization 只做可缓存、稳定的粗粒度规则。
- 细粒度 `user_id`、团队空间、Agent、会话、显式 grant/deny 走 external authorization 调平台 AuthGateway。

### Serverless Recommendation

在允许 K8s/K3s 后，推荐排序调整为：

1. **Knative Serving + Knative Functions**
   适合作为本项目默认 Serverless 运行层。它把用户代码标准化为 OCI image / Knative Service，天然获得 revision、流量切分、scale-to-zero、并发/RPS autoscaling、资源限制和 K8s 网络治理能力。对本项目来说，最关键的是它不会把平台锁死在“函数框架”里：用户上传 Java/Python/JS 代码可以被平台包装成 HTTP function，也可以升级为完整 HTTP service。

2. **Fission**
   适合作为并行 PoC，验证“上传一段代码包即生成函数”的开发体验。Fission 的函数环境、warm pool、executor 模型比 Knative 更像传统 FaaS，对 FDE 快速上传小函数更友好。风险是长期生态和通用容器服务能力不如 Knative，且仍需平台自建 MCP/HTTP contract。

3. **OpenFaaS on Kubernetes**
   如果团队已有 OpenFaaS 运维经验，可以作为备选。它的开发体验成熟，模板丰富，但强 autoscaling 和 scale-to-zero 的生产能力需要注意 CE/Pro 边界；在“开源 + 强弹性伸缩”硬要求下不作为首选。

4. **Apache OpenWhisk**
   如果需求更接近“完整传统 FaaS 平台”，OpenWhisk 能力强，多语言和 action 模型成熟。但它的 Controller/Invoker/Kafka/CouchDB 等组件会显著增加本项目运维负担，不适合作为 K3s 下的最小可交付首选。

5. **OpenFunction / Nuclio / faasd**
   均不作为当前首选。OpenFunction release 偏旧且依赖链较重；Nuclio 更偏专项实时/数据处理场景；faasd 是无 K8s 前提下的轻量 fallback，当前约束变化后优先级下降。

## Route A: 自研 Tools Gateway + 现成 Serverless 平台

### Summary

平台 backend 自研 Tools Gateway / AuthGateway / ToolExecutor / MCP Bridge，Gateway 能力由本项目掌控。Serverless 运行层优先集成 Knative Serving/Functions；Fission 作为 PoC 对照或特定函数体验方案。

```text
Claude Agent SDK
  |
  | mcp_servers = platform tools gateway
  v
Platform MCP Bridge / Tools Gateway (FastAPI)
  |
  | AuthGateway: user_id / role / team_space / grants / acl
  v
ToolExecutor
  |---- MCPProvider      -> external MCP
  |---- HTTPProvider     -> external HTTP API
  |---- UserCodeProvider -> Knative Service / Fission Function
  |---- InternalProvider -> internal APIs
  |
  v
StabilityInterceptor + MetricsCollector
```

### Pros

- 与当前 FastAPI 权限、会话、团队空间、usage 统计贴合。
- 权限模型最容易保持一致，业务审计无需跨系统推导。
- 不依赖 agentgateway 的动态配置、external auth 和 Kubernetes controller 成熟度。
- 可先用 mock provider 或单个 Knative provider 快速闭环。

### Cons

- MCP federation、OpenAPI to MCP、MCP authorization、OAuth 等能力要自研或逐步补。
- 长期会重复 agentgateway 已经做的 MCP/Agent 网关能力。
- 如果后续仍要接 agentgateway，迁移会有一次数据面重构。
- 网关作为 FastAPI 服务承压，需要自行处理长连接、streaming、backpressure 和限流。

### When To Choose Route A

- agentgateway PoC 发现 dynamic route、external auth、MCP policy 或稳定性不满足生产要求。
- 首版只需要少量内部工具，MCP federation/OpenAPI-to-MCP 等能力不急。
- 团队希望把所有权限和审计逻辑留在现有 FastAPI 内，接受后续补齐网关能力。

## Route B: agentgateway + 现成 Serverless 平台

### Summary

agentgateway 作为现成 Gateway 和工具数据面，平台 backend 作为控制面和权限真相。Serverless 运行层优先集成 Knative Serving/Functions；Fission 作为 PoC 对照。

```text
Claude Agent SDK
  |
  | mcp_servers = agentgateway MCP endpoint
  | identity = short-lived JWT / session-bound token
  v
agentgateway
  |
  | MCP authz / HTTP policies / rate limit / timeout / telemetry
  | external auth -> Platform AuthGateway
  v
Backends
  |---- external MCP servers
  |---- OpenAPI / HTTP APIs
  |---- Knative Services / Fission Functions
  |---- internal APIs
```

### Control Plane

```text
Platform FastAPI
  |---- ConnectorRegistry
  |---- ToolACL / UserToolGrant
  |---- ServerlessPackage / BuildJob
  |---- AuthGateway.check()
  |---- GatewayPolicySync
  |
  | sync Gateway API / agentgateway config
  | sync Knative Service / Fission Function
  v
K3s / Kubernetes
  |---- agentgateway
  |---- Knative Serving + Functions
  |---- optional Fission
  |---- namespaces / NetworkPolicy / ResourceQuota
```

### Integration Model

1. 平台创建 connector 后，生成 agentgateway backend/route/policy 配置。
2. 用户上传代码包后，平台创建 build job，产出 OCI image，并部署为 Knative Service 或 Fission Function。
3. Agent 启动时，`runner.py::stream_chat()` 不再注入多个外部 MCP；只注入 agentgateway 暴露的统一 MCP endpoint。
4. 平台按 `user_id`、`agent_id`、`space_id` 编译 `allowed_tools`，只把当前用户可见的 MCP tool 暴露给 Claude Agent SDK。
5. Agent 调用 MCP tool 时，agentgateway 先执行稳定的 MCP authorization。
6. 对需要动态权限的工具，agentgateway 调平台 `AuthGateway.check()`。
7. 平台 AuthGateway 基于 `user_id`、`session_id`、`agent_id`、`space_id`、`connector_id`、`tool_name`、`mcp.tool.arguments` 返回 allow/deny 和上下文注入要求。
8. agentgateway 转发到外部 MCP、HTTP API、Knative Service、Fission Function 或 INTERNAL API。
9. agentgateway 写 OpenTelemetry，平台写业务级 `ToolCallLog`，并与现有 usage 统计按 `trace_id` 打通。

### Pros

- 原生 MCP Gateway，能直接处理 MCP tool、tool arguments、MCP federation、OpenAPI to MCP。
- Kubernetes/Gateway API 支持使其和 Knative/Fission 的 K3s 部署模型一致。
- External authorization 能把权限决策交回本系统，避免双写权限真相。
- 比 APISIX/Kong 更贴合 Agent-to-tool 场景。
- 可减少自研 MCP proxy、OpenAPI-to-tool、MCP policy 的长期负担。

### Cons

- 项目年轻，虽然活跃，但作为生产核心网关需要 PoC 验证稳定性、动态配置和升级策略。
- 本系统当前是 session cookie，接 agentgateway 需要短期 JWT/API key 或 external auth 上下文传递方案。
- agentgateway 仍不能替代 ConnectorRegistry、权限预览、审计日志、UserCode package 管理。
- Knative/Fission 的函数生命周期仍需平台适配，不由 agentgateway 解决。
- 如果在 K3s 内同时引入 agentgateway、Knative、Fission，组件数量会上升，首版应避免三者全量上线。

## Recommendation

当前建议：

1. **Gateway：路线 B 优先，现成 Gateway 选 agentgateway。**
   agentgateway 的 MCP 原生能力正好命中本项目工具平台的核心。APISIX/Kong 不应作为路线 B 首选，除非 agentgateway PoC 在稳定性或权限集成上失败。

2. **Serverless：首选 Knative Serving + Knative Functions on K3s。**
   Knative 更适合作为长期运行层：多语言通过 OCI image/Buildpacks 解决，弹性伸缩和 scale-to-zero 是核心能力，K8s 资源治理也更自然。用户代码统一暴露为 HTTP contract，再由 agentgateway 或平台 MCP Bridge 映射为 MCP tool。

3. **Fission：作为并行 PoC，而不是首选默认底座。**
   Fission 用来验证“代码包函数化”的 FDE 开发体验、warm pool、Java/Python/Node 函数部署速度。如果 PoC 证明 Knative Functions 的上传体验过重，可以在用户代码工具场景引入 Fission，但不要让外部 MCP/API/INTERNAL 能力依赖 Fission。

4. **路线 A：保留为风险兜底。**
   如果 agentgateway 的 external auth、MCP policy、Kubernetes controller 或动态配置无法满足，本项目可以先自研 Tools Gateway，并仍然集成 Knative。这样 Serverless 运行层选型不受网关路线影响。

最终推荐组合：

```text
Preferred:
  agentgateway + Knative Serving/Functions + K3s

PoC comparison:
  agentgateway + Fission + K3s

Fallback:
  自研 Tools Gateway + Knative Serving/Functions + K3s
```

## Shared Platform Responsibilities

无论选路线 A 还是 B，平台仍必须自有这些能力：

- `ConnectorSpec`：工具来源元数据，kind = `mcp_server | http_api | user_code | internal`。
- `ToolACL`：团队、角色、空间维度 ACL。
- `UserToolGrant`：user_id 显式 allow/deny，deny 优先。
- `ToolInvokeContext`：`user_id`、`team_id`、`space_id`、`agent_id`、`session_id`、`trace_id`。
- `ToolCallLog`：业务调用画像，和 agentgateway OpenTelemetry trace_id 打通。
- `ServerlessPackage`：上传包、manifest、语言、构建状态、运行平台、函数/action id。
- `BuildJob`：代码包构建任务，产出 OCI image，记录构建日志和失败原因。
- `RuntimeDeployment`：Knative Service / Fission Function 的运行状态、revision、replica、endpoint。
- `GatewayPolicySync`：把 connector、route、backend、policy 同步到 agentgateway。
- `AuthGateway.check()`：给 agentgateway external authorization 调用的统一鉴权 API。

## User Code Runtime Contract

为了避免被某个 FaaS 产品锁死，用户上传代码建议统一成平台 contract：

```text
GET  /healthz
GET  /__tools/list
POST /__tools/call
```

`/__tools/call` 请求必须包含：

- `tool_name`
- `arguments`
- `context.user_id`
- `context.team_id`
- `context.space_id`
- `context.agent_id`
- `context.session_id`
- `context.trace_id`

平台提供 Python/Java/Node 三种 starter SDK。SDK 负责解析平台上下文、暴露工具清单、做参数校验，并把业务返回转换成 MCP/HTTP 统一响应。

这个 contract 使 Knative 和 Fission 都能接入：

- Knative：每个包构建成 OCI image，部署为 Knative Service。
- Fission：每个包部署为 function/environment，再由 adapter 暴露同样的 HTTP contract。
- agentgateway：只看统一 HTTP endpoint 或 MCP bridge，不关心底层运行平台。

## K3s Runtime Security Baseline

用户代码运行层至少需要这些基线：

- namespace 按团队或项目空间隔离，避免所有用户代码共用默认命名空间。
- 每个 connector 使用独立 ServiceAccount，不允许挂载 backend 的权限。
- 默认拒绝出向流量，只按 connector manifest 放行白名单域名/IP。
- ResourceQuota 和 LimitRange 限制 CPU、内存、Pod 数、临时存储。
- PodSecurity 设为 restricted：非 root、禁止 privileged、禁止 hostPath、只读根文件系统。
- 使用 seccomp/AppArmor 默认 profile。
- Secret 只通过平台托管凭据注入，不允许用户包自带明文密钥。
- 构建和运行分离，BuildJob 无法访问运行时敏感 Secret。
- 每次调用带 `trace_id`，调用日志、网关日志、K8s event 可串联排查。

## What Changes In The PRD / Tech Design

需要调整原 `tools-platform-tech-design.md` 的表述：

- `现成 Gateway` 首选从 APISIX/Kong 调整为 **agentgateway**。
- `Serverless 沙箱` 改为 `Serverless/FaaS 执行平台 + K3s 安全基线`。
- 首选 Serverless 候选从 OpenWhisk/faasd 调整为 **Knative Serving + Knative Functions**。
- 新增 **Fission PoC**，用于验证传统 FaaS 上传体验、warm pool 和 Java/Python/Node 函数部署。
- `Agent 直连 MCP/API` 改为 `Agent 调用 agentgateway 或平台 MCP Bridge 暴露的工具入口`。
- 新增 `GatewayPolicySync`、`ExternalAuthAdapter`、`ServerlessProvider`、`BuildJob`、`RuntimeDeployment` 五个设计组件。
- 明确 K8s/K3s 只是运行与隔离基础设施，不替代平台权限模型。

## Decision Criteria

PoC 后按这些标准冻结方案：

| 标准 | agentgateway + Knative | agentgateway + Fission | 自研 Gateway + Knative |
| ---- | ---------------------- | ---------------------- | ---------------------- |
| MCP 原生能力 | 高 | 高 | 需自研 |
| 权限与本系统打通 | 依赖 external auth PoC | 依赖 external auth PoC | 最高 |
| 多语言支持 | 高，容器/Buildpacks 路线 | 高，函数环境路线 | 高，取决于 Knative |
| 弹性伸缩/scale-to-zero | 高 | 中高，需实测 | 高 |
| FDE 上传代码体验 | 中，需要平台包装 | 高 | 中，需要平台包装 |
| 运维复杂度 | 中 | 中高 | 中 |
| 长期可扩展性 | 高 | 中 | 中 |

冻结规则：

- 如果 agentgateway external auth + MCP tool policy 跑通，路线 B 胜出。
- 如果 Knative 三语言样例的构建、冷启动、scale-to-zero 和网络隔离都达标，Knative 做默认 Serverless。
- 如果 Fission 的上传体验显著优于 Knative，并且 autoscaling/隔离达标，可把 Fission 作为 `user_code` provider，但外部 MCP/API/INTERNAL 仍走 agentgateway。
- 如果 agentgateway PoC 失败，不影响 Knative 作为 Serverless 运行层，切换为路线 A。

## Open Questions

1. K3s 是作为平台默认依赖引入，还是只在需要用户代码托管时启用？
   推荐默认引入 K3s，但允许纯外部 MCP/API 场景禁用 Serverless 运行层。

2. 用户代码是否必须统一 HTTP contract？
   推荐必须统一，否则 Gateway、审计、降级和 SDK 会被 Knative/Fission/OpenWhisk 的差异绑死。

3. 构建链路选 Buildpacks、Dockerfile、还是平台模板？
   推荐首版用平台模板 + Buildpacks，限制自由 Dockerfile，降低供应链和逃逸风险。

4. agentgateway 配置使用 Kubernetes Gateway API/CRD，还是平台直接调用 agentgateway 管理 API？
   推荐 K3s 场景优先 Gateway API/CRD，便于 GitOps 和状态观测；本地开发可生成静态配置。

5. scale-to-zero 的冷启动目标是多少？
   推荐 P0 不把冷启动延迟作为硬指标，只要求可恢复；P1 再按高频工具设置 `minScale=1` 或 warm pool。

## Current Recommendation

**Gateway 推荐 agentgateway。Serverless 平台推荐 Knative Serving + Knative Functions on K3s。Fission 作为并行 PoC，用来验证用户上传代码的函数体验。**

更具体地说：本系统的权限真相和控制面仍在 FastAPI；agentgateway 负责 MCP/HTTP 数据面和协议治理；Knative 负责用户代码的弹性运行；Fission 只在 PoC 证明其上传体验明显优于 Knative 时，作为 `user_code` 的补充 provider。
